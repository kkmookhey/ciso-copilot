## Overview

The AI-security subsystem scans a customer's GitHub repos for AI/LLM
usage and turns what it finds into entities + edges (an **AI-BOM**) and
findings in the same unified `entities` / `edges` / `findings` tables
used by cloud/identity scanning (AI is a lens, not a silo). It is five
Lambda directories plus one empty placeholder:

- **`ai_github/`** ‚Äî REST API for the GitHub App connect flow (install
  URL ‚Üí complete ‚Üí list repos ‚Üí delete). API Gateway, Cognito-auth'd.
- **`ai_scanner/`** ‚Äî the container Lambda that does the work. SQS-driven
  (`ai-scan-queue`): clone repo ‚Üí run 9 deterministic detectors +
  correlator + Trivy SCA ‚Üí commit transactionally ‚Üí enqueue the matcher.
- **`ai_supply_chain_matcher/`** ‚Äî SQS-driven. Joins Trivy `sca_vuln`
  findings with the AI-BOM graph + KEV threat indicators ‚Üí emits
  `ai_supply_chain_active` CRITICAL findings + APNs push.
- **`ai_summary/`** ‚Äî `GET /ai/summary` read API: aggregates AI-touching
  findings for the `/ai` web view (score, by-source, by-framework,
  top-people).
- **`ai_scan_api/`** ‚Äî **EMPTY / DEAD.** Only stray `__pycache__` dirs;
  not git-tracked. Its job ("start a scan") was absorbed into
  `entities_api` (see `entities_api/main.py:14` "Replaces ai_scan_api").

The scan-trigger entry point is **NOT in this subsystem**: `POST
/ai/scans` lives in `platform/lambda/entities_api/main.py:69` and is what
enqueues `ai-scan-queue`. CDK wiring is in `platform/lib/scan-stack.ts`
(`aiScanQueue`, `aiScanner`, `matcherQueue`, `matcherFn`).

### Pipeline, end to end

```
web ‚Üí POST /ai/connections/github/install_url  (ai_github)
    ‚Üí GitHub install ‚Üí POST /ai/connections/github/complete (ai_github)  ‚Üí ai_connections row
    ‚Üí GET /ai/connections/{id}/repos  (ai_github)
web ‚Üí POST /ai/scans  (entities_api)  ‚Üí INSERT ai_scans (status=queued) ‚Üí SQS ai-scan-queue
                                       ‚Üí ai_scanner.handler
                                            clone ‚Üí 9 detectors ‚Üí correlator ‚Üí Trivy
                                            ‚Üí unified_writer.commit_scan (entities/edges/findings + framework registry)
                                            ‚Üí SQS ai-supply-chain-matcher-queue
                                       ‚Üí ai_supply_chain_matcher.handler  ‚Üí ai_supply_chain_active + APNs push
web ‚Üí GET /ai/summary  (ai_summary)   ‚Üí aggregated AI-touching view
```

## THE DETECTOR INVENTORY (headline artifact)

Nine detectors run per scan, in this order
(`ai_scanner/main.py:38-41` + `crossdomain`), then the correlator. Each
is a module exposing `detector_id`, `detector_version`, and
`detect(ctx) -> DetectorResult`. They emit `EntityEmission` /
`EdgeEmission` / `FindingEmission` (shapes in
`detectors/base.py:14-66`). All evidence packets are built by
`evidence.build()` (`evidence.py:15`), `model=None` (deterministic, no
LLM).

**CRITICAL FRAMEWORK-MAPPING FACT:** the AI **code** detectors below do
**NOT** set `frameworks` on their findings, and **no registry rule
matches their `check_id`s** (verified: 0 hits grepping the four code
finding-types + `sca_vuln` against `ai_framework_registry.json`). So
`mcp_with_broad_perms`, `autonomous_loop_no_human_in_loop`,
`prompt_with_secret_pattern`, `hardcoded_credential_in_ai_module`, and
`sca_vuln:*` ship with `frameworks={}`. They still surface in `/ai`
because `ai_summary._IS_AI_TOUCHING` also matches on `e.domain='ai'` /
AI entity kind, **not only** framework keys. The NIST AI RMF / OWASP LLM
Top 10 mappings that DO exist (registry rules) apply to the **cloud-side**
AI checks (Bedrock/SageMaker/Entra-signin), not to these code findings.
See the registry table further down.

### 1. `framework` ‚Äî `detectors/framework.py`
- **detector_id** `ai.detectors.framework` v0.2.0
- **Detects:** imports of AI frameworks via ripgrep on `.py`:
  `langchain, langgraph, llama_index, llama_cpp, crewai, autogen,
  semantic_kernel, dspy` (`framework.py:13-16`).
- **Emits:** `ai_framework` entity (natural_key = bare framework name,
  cross-repo deduped) + `github_repo --uses--> ai_framework` edge
  (`framework.py:45-67`). **No findings.**
- **Frameworks mapped:** none at emit time.

### 2. `model_usage` ‚Äî `detectors/model_usage.py`
- **detector_id** `ai.detectors.model_usage` v0.3.0
- **Detects:** LLM model IDs in `.py`. A file must carry a provider
  signal (SDK import or API URL ‚Äî `model_usage.py:26-36`) then match
  `model="..."` kwarg-style **and** `"model": "..."` JSON-style literals.
  Provider resolved from model-id hint first (`anthropic.claude*`‚Üíbedrock,
  `claude-*`‚Üíanthropic, `gpt-*`‚Üíopenai; `model_usage.py:42-46,144`).
- **Emits:** `ai_model` entity (natural_key `{provider}/{model_id}`,
  cross-repo deduped) + `github_repo --calls--> ai_model` edge. **No
  findings.**
- **Frameworks mapped:** none at emit time.

### 3. `mcp_server` ‚Äî `detectors/mcp_server.py`
- **detector_id** `ai.detectors.mcp_server` v0.2.0
- **Detects:** MCP servers two ways ‚Äî (a) Python AST: `from mcp.server
  import Server` + `Server("name")` + `@<var>.list_tools()` returning
  list-of-dicts with `"name"` (`mcp_server.py:70-145,212`); (b) config
  files `mcp.json` / `claude_desktop_config.json` with `mcpServers`
  mapping (`mcp_server.py:247`).
- **Emits:** `ai_mcp_server` entity (per-file nk
  `{repo_nk}::{path}::{server}`), `ai_tool` entity per declared tool
  (Python path only), `github_repo --deploys--> ai_mcp_server` edge,
  `ai_mcp_server --invokes--> ai_tool` edge.
- **Finding:** `mcp_with_broad_perms` (**HIGH**) when any tool name
  starts with `create_/delete_/write_/update_` (`mcp_server.py:35,177`).
- **Frameworks mapped:** none.

### 4. `agentic_workflow` ‚Äî `detectors/agentic_workflow.py`
- **detector_id** `ai.detectors.agentic_workflow` v0.2.0
- **Detects (heuristic, AST):** a function with ALL of ‚Äî (1) a `while`
  loop or recursive self-call, (2) an LLM `.create()` / `.invoke_model()`
  call in a module that imports an LLM SDK, (3) a tool-dispatch token
  (`tool_calls/tool_call/function_call`) (`agentic_workflow.py:62-100`).
  `confidence='medium'`, false positives expected.
- **Emits:** `ai_agent` entity (nk `{repo_nk}::{path}::{fn}`). **No
  agent‚Üímodel edge here** ‚Äî the correlator emits that.
- **Finding:** `autonomous_loop_no_human_in_loop` (**MEDIUM**) per agent.
- **Frameworks mapped:** none.

### 5. `vector_db` ‚Äî `detectors/vector_db.py`
- **detector_id** `ai.detectors.vector_db` v0.2.0
- **Detects:** Python imports of `chromadb, pinecone, weaviate,
  qdrant_client, faiss` (`vector_db.py:22-28`) + pgvector via
  `CREATE EXTENSION ... vector` in any `.sql` (`vector_db.py:92`).
- **Emits:** `ai_vector_db` entity (bare-name nk, cross-repo deduped) +
  `github_repo --retrieves--> ai_vector_db` edge. **No findings.**
- **Frameworks mapped:** none.

### 6. `embedding` ‚Äî `detectors/embedding.py`
- **detector_id** `ai.detectors.embedding` v0.2.0
- **Detects:** OpenAI `model="text-embedding-*"`, plus Voyage/Cohere
  `.embed(` calls with provider context within 100 chars
  (`embedding.py:20-23,41-61`).
- **Emits:** `ai_embedding` entity (nk `{provider}/{model_id}`,
  e.g. `openai/text-embedding-3-small`, `voyage/embed`, `cohere/embed`)
  + `github_repo --generates--> ai_embedding` edge. **No findings.**
- **Frameworks mapped:** none.

### 7. `prompt` ‚Äî `detectors/prompt.py`
- **detector_id** `ai.detectors.prompt` v0.2.0
- **Detects:** prompt files (under `prompts/`, `prompt*.{txt,md}`,
  `*.prompt`) + inline strings >200 chars passed as
  `prompt=/system=/instructions=` kwargs to `.create(...)`
  (`prompt.py:33-34,78-112`).
- **Emits:** `ai_prompt` entity (per-file nk) +
  `github_repo --accesses--> ai_prompt` edge.
- **Finding:** `prompt_with_secret_pattern` (**HIGH**) when the prompt
  body matches a secret regex (openai `sk-`, anthropic `sk-ant-`, slack
  `xoxb-`, aws `AKIA`; `prompt.py:36-41`).
- **Frameworks mapped:** none.

### 8. `secrets_in_ai_code` ‚Äî `detectors/secrets_in_ai_code.py`
- **detector_id** `ai.detectors.secrets_in_ai_code` v0.2.0
- **Detects:** the same four secret regexes, but **only** in `.py` files
  that import an LLM SDK (`from openai/anthropic`, `bedrock-runtime`).
  Correlation gate: a generic secret in a non-AI file is out of scope.
- **Emits:** **no entities, no edges.** One
  `hardcoded_credential_in_ai_module` finding (**HIGH**) per match,
  linked via legacy free-text `subject_type='ai_module'` / `subject_ref`
  (no entity FK; `secrets_in_ai_code.py:77-92`).
- **Frameworks mapped:** none.

### 9. `crossdomain` ‚Äî `detectors/crossdomain.py`
- **detector_id** `ai.detectors.crossdomain` v0.1.0
- **Detects:** AWS OIDC `role-to-assume: arn:aws:iam::...:role/...` in
  `.github/workflows/*.{yml,yaml}` (`crossdomain.py:24-27`).
- **Emits:** `github_repo --deploys_to--> aws_iam_role` edge only
  (`confidence='medium'`). The IAM-role target may be a stub until the
  cloud scanner hydrates it. **No entities, no findings.**

### Correlator (runs after the 9) ‚Äî `detectors/correlator.py`
- **detector_id** `ai.detectors.correlator` v0.2.0
- Adds **derived edges** from same-file colocation of entities
  (`correlate(ctx, results)`):
  - `ai_agent` + `ai_mcp_server` in a file ‚Üí `ai_agent --invokes-->
    ai_mcp_server`
  - `ai_agent` + `ai_model` ‚Üí `ai_agent --orchestrates--> ai_model`
  - `ai_model` + `ai_vector_db` + `ai_prompt` (RAG shape) ‚Üí `ai_model
    --retrieves--> ai_vector_db`
- **No entities, no findings.** Reproduces upstream natural-key formulas
  by hand (must stay in sync; `correlator.py:13-18`).

### Trivy SCA pass (not a detector module) ‚Äî `ai_scanner/trivy.py`
- Run inline by `main.py:108` after detectors, capped at 60s.
- `trivy fs --severity HIGH,CRITICAL --scanners vuln` ‚Üí one
  `sca_vuln:{CVE}` finding per vuln, subject = `github_repo`
  (`trivy.py:65-88`). Evidence packet carries `package/version/cve/
  fixed_version/manifest`. **No framework mapping.** These are the rows
  the supply-chain matcher consumes.

## AI-BOM / AI inventory: how it's built

The AI-BOM is the set of `entities` (kind `ai_*` / `github_repo`) and
`edges` written by `unified_writer.commit_scan` (`unified_writer.py:44`)
in **one transaction**. Read back through `entities_api`
(`/entities`, `/entities/{id}/graph`, `/relationships`) and aggregated
by `ai_summary`.

**Entity kinds emitted by this subsystem:** `github_repo` (the root,
emitted by `main.py:90` not by a detector), `ai_framework`, `ai_model`,
`ai_mcp_server`, `ai_tool`, `ai_agent`, `ai_vector_db`, `ai_embedding`,
`ai_prompt`. Cross-domain stub target: `aws_iam_role`.

**Edge kinds:** `uses` (repo‚Üíframework), `calls` (repo‚Üímodel),
`deploys` (repo‚Üímcp_server), `invokes` (mcp_server‚Üítool, agent‚Üímcp),
`retrieves` (repo‚Üívector_db, model‚Üívector_db), `generates`
(repo‚Üíembedding), `accesses` (repo‚Üíprompt), `orchestrates`
(agent‚Üímodel), `deploys_to` (repo‚Üíiam_role).

**Writer mechanics (`unified_writer.py`):**
- Entity UPSERT on `ON CONFLICT (tenant_id, kind, natural_key)`; returns
  the **persisted** id (existing on conflict). Always uses returned id
  (`_upsert_entity`, line 178; doc at 9-13).
- Edges resolve src/tgt by `(kind, natural_key)`; if a target isn't in
  this scan, a `{_stub:true}` stub entity is created
  (`_resolve_or_stub`, line 258).
- Findings UPSERT on natural key
  `(tenant_id, conn_id, check_id, COALESCE(resource_arn,''),
  COALESCE(region,''))` ‚Äî re-scans refresh in place, preserve
  `first_seen`, clear `resolved_at` (`_insert_finding`, line 317).
- Each finding passes through the **framework registry**
  (`framework_registry.apply`) before insert (line 105).
- Post-commit, every CRITICAL+fail finding fans out to Slack via
  `broadcast_fanout.publish_if_critical` (line 146) ‚Äî done AFTER commit
  to avoid the subscriber racing an uncommitted row (comment at 84-89).

**SCHEMA GOTCHA (matches CLAUDE.md):** the AI graph emits **`github_repo
--uses--> ai_framework` directly**. There is **no `ai_agent`
intermediate** between repo and framework. The supply-chain matcher's SQL
joins this correctly (`e.kind = 'uses'`, repo = `e.source_entity_id`,
matcher `main.py:66-72`), even though the matcher's **comment** at
`main.py:45-48` still describes a stale `ai_framework ‚Üí ai_agent` /
`agent.parent_id` shape that does not exist. The code is right; the
comment lies. (`entities` has no `parent_id` column.)

## The GitHub App (`ai_github/`)

REST API behind API Gateway `/v1/ai/connections/...`, Cognito-auth'd,
tenant resolved via `users.sso_subject` with the `identities[0].userId`
‚Üí `sub` fallback (`helpers.py:46`). **It does not trigger scans.**

- `github_app.py` ‚Äî GitHub App client: mints RS256 app JWT (10-min,
  `iss` = client_id or app_id; line 39), exchanges installation tokens
  (1hr, cached 50min per warm container; line 61), lists installation
  repos (line 85), revokes tokens (line 106).
- `state_jwt.py` ‚Äî stdlib HS256 short-lived (5min) state JWTs for the
  OAuth-ish install round-trip (sign/verify, secret from Secrets Manager).
- `main.py` routes:
  - `POST /ai/connections/github/install_url` ‚Äî sign state, return
    `https://github.com/apps/{slug}/installations/new?state=...`.
  - `POST /ai/connections/github/complete` ‚Äî verify state, validate
    installation via app JWT, UPSERT `ai_connections` on `(tenant_id,
    provider, github_installation_id)`, return **persisted** id (the
    returned-vs-local-UUID bug fix, comment line 118-123).
  - `GET /ai/connections` ‚Äî list non-revoked connections.
  - `GET /ai/connections/{id}/repos` ‚Äî page installation repos; GitHub
    401 ‚Üí mark connection `revoked`, return 409.
  - `DELETE /ai/connections/{id}` ‚Äî best-effort token revoke + mark
    `revoked`.

**There is no webhook handler.** Scans are user-initiated (`POST
/ai/scans` in `entities_api`), not push/webhook-triggered. A spec
proposing "wire up the GitHub webhook to auto-scan on push" would be
proposing something genuinely new.

## Supply-chain matcher / scan API / summary roles

### `ai_supply_chain_matcher/`
- SQS-triggered (`ai-supply-chain-matcher-queue`), one message per scan,
  enqueued by `ai_scanner/main.py:127`.
- `_MATCH_SQL` (`main.py:52`) joins `findings` (sca_vuln) ‚Üí `entities`
  (ai_framework, matched by `LOWER(display_name)=LOWER(package)`) ‚Üí
  `edges` (kind `uses`) ‚Üí `entities` (github_repo) ‚Üí `threat_indicators`
  (kind `cve`, source `kev`, column `indicator_value`). Dedup guard via
  `NOT EXISTS` on prior `ai_supply_chain_active`.
- Emits `ai_supply_chain_active:{cve}:{repo_id}` finding at **CRITICAL**,
  `conn_id` resolved from `scans.conn_id` (FK satisfaction;
  `_conn_id_for_scan`, line 143) or nil-UUID, `frameworks=[]` (note:
  written as JSON array `'[]'` here, line 193 ‚Äî the rest of the system
  uses `{}` objects; works because this finding never hits the
  `?|` framework filter path, but it is inconsistent).
- Fires APNs push via the shared `_shared/push.py`.
- **Build note:** there are two identical copies of the source ‚Äî
  `main.py` and `build/main.py` (build/ is what `build.sh` vendors into
  `dist/matcher.zip`, deployed as a **zip** Lambda per
  `scan-stack.ts:364-371`). The matcher is the only non-container AI
  Lambda.

### `ai_scan_api/` ‚Äî DEAD placeholder
Empty directory (only `__pycache__`, not git-tracked). Functionality
moved to `entities_api`. Do not build here.

### `ai_summary/`
- `GET /ai/summary` read API for the `/ai` web view.
- `_IS_AI_TOUCHING` SQL predicate (`main.py:70`): a finding is AI-touching
  if (frameworks `?|` any of the 8 AI framework keys) OR (subject entity
  `domain='ai'`) OR (entity kind in `_AI_RESOURCE_KINDS`) OR
  (`evidence_packet->>'is_ai'='true'`). This predicate is **mirrored** in
  `framework_registry._is_ai_touching` and `_AI_RESOURCE_KINDS` ‚Äî three
  copies of the same allowlist (kept in code "so deploys carry the
  truth", but a drift risk).
- Returns `score`, `by_source` (aws/azure/code/entra), `by_framework`
  (per the 8 AI frameworks, per status), `top_people` (by
  `commit_author_email`/`iam_owner_email`/`entra_upn` in evidence
  packets), and `frameworks_meta` from `framework_meta.ai_family_meta()`.

## Framework registry ‚Äî WHAT ALREADY EXISTS (the anti-rebuild section)

`ai_framework_registry.json` + the `framework_registry.py` engine are the
CME-v2 two-stage compliance crosswalk (normalize ‚Üí augment;
`framework_registry.apply`, line 264). Applied to **every** finding in
`unified_writer.commit_scan`.

**8 AI-family framework packs already shipped** (registry `frameworks`
block; `family='ai'`):

| key | name | rewrite_rules |
|---|---|---|
| `nist_ai_rmf` | NIST AI RMF | 19 |
| `iso_42001` | ISO/IEC 42001 | 0 |
| `soc2_ai` | SOC 2 + AI | 0 |
| `eu_ai_act` | EU AI Act | 8 |
| `nist_ai_600_1` | NIST AI 600-1 | 12 |
| `owasp_llm_top10` | OWASP LLM Top 10 | 10 |
| `owasp_agentic` | OWASP Agentic | 0 |
| `mitre_atlas` | MITRE ATLAS | 16 |

**13 mapping rules already shipped** (registry `rules`), keyed on
`check_id_eq` for Entra AI-signin tiers (`ai_signin_personal/corp/
unknown_tier`) and cloud AI baseline checks (`bedrock-content-filter`,
`bedrock-vpc-endpoint`, `cloudtrail-ai-events`,
`s3-training-data-versioned`, 7√ó `sagemaker-*`). Each fans out into
2‚Äì5 of the AI framework packs above.

**Engine capabilities that already exist** (`framework_registry.py`):
selectors `check_id_eq / check_id_glob / domain / resource_type_glob /
ai_touching / evidence_packet_eq` (line 14-21); per-framework
`rewrite_rules` to canonicalize control IDs (`_normalize_stage`, line
192); additive set-union merge with `_registry_rule_ids` provenance
(`_augment_stage`, line 236); load-time schema validation that fails
cold-start on malformed registry (`validate_registry`, line 60).

## Dead code (suspected)

1. **`ai_scan_api/` ‚Äî whole directory.** **Confidence: High.** Empty
   (only `__pycache__`), not git-tracked (`git ls-files` returns
   nothing). `entities_api/main.py:14` says "Replaces ai_scan_api". No
   CDK reference. Safe to delete.

2. **`unified_writer.py:249` ‚Äî the bound `stub` SQL param.**
   **Confidence: High.** Bound as `{"name":"stub",
   "value":{"booleanValue":...}}` but the token `:stub` never appears in
   the INSERT SQL text (verified: 0 occurrences of `:stub` in any SQL
   block; the only `stub` SQL logic is the `EXCLUDED.attributes - '_stub'`
   on conflict). RDS Data API silently ignores the extra param. The
   `stub` function arg still drives the `attributes` JSON content
   (line 194), so only the **param binding** is dead, not the arg.

3. **`EntityEmission.connection_id` (`base.py:25`) as consumed by the
   writer.** **Confidence: Med.** Every detector sets it, but
   `unified_writer` never reads `e.connection_id` ‚Äî it uses
   `ctx.connection_id` for the findings `conn_id` (line 351) and never
   stores a connection on entities at all. Grep: `connection_id=` is
   written by all detectors, read nowhere downstream. Likely vestigial
   from a pre-SP1 shape. (Same for `EntityEmission.source_path`, but that

## Optimization opportunities

1. **Per-file full re-read across detectors.** `model_usage`,
   `embedding`, `mcp_server`, `agentic_workflow`, `prompt`,
   `secrets_in_ai_code` each independently `rglob("*.py")` and
   `read_text()` every `.py` file (6 full walks + reads of the same
   tree). On a large repo this is the dominant cost. A shared
   pre-pass that reads each file once (and parses one AST per file,
   reused by the 3 AST detectors) would cut I/O and parse work ~6√ó.

2. **`ai_summary` runs 4 separate full-table aggregate queries** per
   request (`_query_score/_by_source/_by_framework/_top_people`,
   `main.py:88-91`), each re-scanning `findings` with the same
   `_IS_AI_TOUCHING` predicate and the same LEFT JOINs. Could be one
   query with grouping sets / CTEs, or a materialized per-tenant rollup
   if `/ai` is hit often.

3. **Matcher re-scans ALL of a tenant's `sca_vuln` findings on every
   message** (`_MATCH_SQL` filters only by `tenant_id`, not `scan_id`),
   so cost grows with cumulative finding count, not per-scan delta. If a
   tenant accumulates many sca_vuln rows this becomes an N√óN join each
   scan. Scoping the match to the triggering `scan_id` would bound it.

## Gotchas / non-obvious behavior (a spec author MUST know)

- **NIST AI RMF + OWASP LLM Top 10 + 6 more AI packs already exist** as
  framework packs with rewrite rules (table above). Do NOT propose
  adding them. This is the exact mistake commit `225e4d8` fixed.
- **AI code-scan findings carry empty `frameworks={}`** ‚Äî no registry
  rule maps the four code finding-types or `sca_vuln`. The framework
  mappings that exist are for **cloud-side** AI checks
  (Bedrock/SageMaker/Entra). If you want code findings tagged to NIST AI
  RMF / OWASP, that is a real gap to fill (add registry rules), not a
  rebuild.
- **`github_repo --uses--> ai_framework` is direct; no `ai_agent`
  intermediate.** The matcher's docstring lies about this; the SQL is
  correct. `entities` has no `parent_id`.
- **The scan trigger is `POST /ai/scans` in `entities_api`,** not in
  `ai_github` and not in the (dead) `ai_scan_api`. `ai_github` only does
  the connect/list/delete flow.
- **No GitHub webhook exists.** Scans are user-initiated only.
- **9 detectors run, not "8".** Docstrings/comments say "eight
  deterministic detectors" (`main.py:6`, `correlator.py:2`) but
  `crossdomain` is a 9th in the `DETECTORS` list (`main.py:38-41`); the
  correlator then runs separately on top.
- **Findings UPSERT in place on re-scan** (natural key incl. check_id +
  resource_arn + region). Detectors that want distinct rows encode
  uniqueness into `finding_type`/`check_id` (Trivy does
  `sca_vuln:{cve}`, matcher does `ai_supply_chain_active:{cve}:{repo}`).
- **`frameworks` must be a JSONB object `{}`,** never an array ‚Äî the iOS
  decoder rejects arrays. The matcher writes `'[]'` (array) at
  `main.py:193`; it works only because that finding never traverses the
  framework UI path, but new code should use `{}`.
- **CRITICAL findings broadcast to Slack post-commit** via
  `broadcast_fanout` ‚Äî wired through `AUTONOMOUS_BROADCAST_QUEUE_URL`.
  Don't re-implement critical-fan-out; reuse `publish_if_critical`.
- **Three copies of the AI-resource-kind allowlist** exist
  (`ai_summary._AI_RESOURCE_KINDS`,
  `framework_registry._AI_RESOURCE_KINDS`,
  `framework_meta.FRAMEWORK_META`). Update all three together.
- **Container-Lambda flat bundling:** ai_scanner imports detectors as
  `from detectors import ...` (package preserved via Dockerfile COPY).
  The matcher is a **zip** Lambda built by `build.sh` into
  `dist/matcher.zip` from the `build/` dir ‚Äî `cdk deploy` requires
  running `build.sh` first.
- **Tenant resolution everywhere uses `identities[0].userId` ‚Üí
  `sub`** fallback (`helpers.py:46`, `ai_summary.py:250`). Reaching for
  `claims.get("sub")` directly 401s federated users.
