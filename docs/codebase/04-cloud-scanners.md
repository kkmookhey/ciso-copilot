# Per-cloud scanner runners (Azure / GCP / Entra) — Codebase Documentation

> Generated by codebase-documentation sweep 2026-06-05. Purpose: give a
> future spec author a grep-able answer to "does this cloud already have a
> scanner / this collector already exist?" before they propose building it.
> Cite `file:line` everywhere.
>
> Scope: the three per-cloud Fargate/Lambda scanner runners under
> `platform/lambda/`. The AWS runner (`shasta_runner/`) and the AI scanner
> (`ai_scanner/`) are the source-of-truth siblings and are referenced but
> not exhaustively documented here. Vendored Shasta / Azure / Google SDK
> internals are out of scope — we document how Shasta checks are *invoked*,
> not what they do.

## Overview

Three runners scan a customer's non-AWS cloud footprint, each wrapping
Shasta's cloud check modules and writing into the same unified
`entities` / `edges` / `findings` / `scans` model the AWS scanner uses:

- **`shasta_runner_azure/`** — Azure subscriptions. Runs as an **ECS
  Fargate task** (`run.py` → `main.handler`). Auth: one service principal
  per connection. Three-stage pipeline (subscription discovery → footprint
  probe → tier-aware parallel scan).
- **`shasta_runner_gcp/`** — GCP projects. Runs as an **ECS Fargate task**
  (`run.py` → `main.handler`). Auth: Workload Identity Federation (no key
  on disk). Same three-stage pipeline; adds an org-mode project
  enumeration branch.
- **`shasta_runner_entra/`** — Microsoft Entra (Azure AD) tenant. Runs as
  a **Lambda container** (`main.handler`). Auth: our multi-tenant app
  registration's client credentials, token-scoped to the customer tenant
  via Microsoft Graph. Single-pass (no tiers, no parallel ScanUnits).

Triggers (verified `platform/lib/scan-stack.ts`, `api-stack.ts`,
`connections_list/main.py`, `onboarding_{azure,gcp}_complete/main.py`):

- Azure scan: `ecs.run_task` from `onboarding_azure_complete` (first scan)
  and `connections_list` (rescan). Task def `azureScanTaskDef`
  (scan-stack.ts). Container override env carries the per-scan params.
- GCP scan: `ecs.run_task` from `onboarding_gcp_complete` and
  `connections_list`. ECS task role `gcpScannerRole`.
- Entra scan: Lambda `ciso-copilot-shasta-runner-entra`
  (scan-stack.ts:210), `DockerImageFunction`. Invoked directly (Lambda
  invoke, not RunTask).

**Azure vs GCP vs Entra are near-clones at the orchestration layer;
Entra is the odd one out** — it does NOT use the shared
`scan_pipeline` / `scan_state` / `unified_writer` modules and writes
findings with its own inline SQL.

## Components

### Azure runner — `platform/lambda/shasta_runner_azure/`

#### `app/main.py`
- **Purpose:** Azure scan orchestrator — credentials, 3-stage pipeline,
  two-phase Quick early-commit.
- **Entry point / handler:** `handler(event, context)` (main.py:97).
- **Key symbols:**
  - `AZURE_MODULES` (main.py:69) — module-name → Shasta
    `run_all_azure_*_checks` fn. 12 modules: iam, governance, storage,
    networking, compute, encryption, databases, appservice, monitoring,
    backup, diagnostic_settings, private_endpoints. **"ai" is wired
    separately** (not in this dict).
  - `CloudScanContext` (main.py:88) — frozen dataclass holding
    `scan_id`/`tenant_id`/`connection_id`/`scanner_version`, read by
    `unified_writer.commit_scan` by attribute.
  - `_build_units()` (main.py:202) — one `ScanUnit` per
    (subscription, module). `name == "ai"` → `_ai_unit`, else
    `_module_unit`.
  - `_ai_unit()` (main.py:225) / `_module_unit()` (main.py:241) — build
    per-unit run callables; each builds a **fresh** `AzureClient` via
    `base_client.for_subscription(sub_id)` + `validate_credentials()`
    inside `run` so concurrent threads never share a mutable SDK client.
  - `_absorb()` (main.py:256) — merge `UnitResults` into accumulators +
    coverage map (unit name format `<sub_id>/<module>`).
  - `_record_subscription_names()` (main.py:274) — best-effort
    `jsonb_set` of `cloud_connections.scope.subscription_names` so the
    web subscription picker shows display names. Consumed by
    `web/src/scan/AzureScanCardBody.tsx:13`.
- **Reads:** Secrets Manager (SP secret by `secret_arn`); env
  `DB_CLUSTER_ARN`/`DB_SECRET_ARN`/`DB_NAME`. Shasta Azure SDK calls.
- **Writes:** `scans` (via scan_state), `entities`/`edges`/`findings`/
  `ai_scans` (via unified_writer.commit_scan), `cloud_connections.scope`.
- **Called by:** `run.py` (Fargate) or direct Lambda invoke.

#### `app/run.py`
- **Purpose:** Fargate entrypoint — reads env vars into the event shape.
- **Key symbols:** `build_event(env)` (run.py:25) maps env → event;
  `_REQUIRED` (run.py:21) = SCAN_ID, TENANT_ID, CONN_ID, AZURE_TENANT_ID,
  CLIENT_ID, SECRET_ARN, SUBSCRIPTION_IDS. `SUBSCRIPTION_IDS` is
  comma-separated; `SCAN_TIER` defaults `quick`. `from main import
  handler` is deferred into `main()` to keep `build_event` test-collectible.

#### `app/azure_credential.py`
- **Purpose:** SP credential injection.
- **Key symbol:** `apply_sp_credentials(secret)` (azure_credential.py:15)
  — writes `AZURE_CLIENT_ID`/`AZURE_CLIENT_SECRET`/`AZURE_TENANT_ID` into
  `os.environ` once at process start for `DefaultAzureCredential`.
  Constant for the whole run, so safe despite parallel threads.

#### `app/subscription_discovery.py`
- **Purpose:** Stage 1 + 2 — footprint probe per subscription.
- **Key symbol:** `discover_subscriptions(subscription_ids, probe)`
  (subscription_discovery.py:24) — parallel probe; any exception or
  non-`active`/`empty` value → `unknown` (never silently dropped, never
  mislabelled `empty`). Probe callable injected from main.py
  (`for_subscription` → `validate_credentials` → `discover_services`).

#### `app/azure_units.py`
- **Purpose:** tier → module-name lists (pure data).
- **Key symbol:** `modules_for_tier(tier)` (azure_units.py:22). Quick =
  2-phase (`iam,governance` / `storage,networking,compute,encryption`);
  Medium = Quick + `databases,appservice,monitoring,ai`; Deep = Medium +
  `backup,diagnostic_settings,private_endpoints`. **The AI pass runs at
  Medium+Deep for Azure** ("ai" in `_MEDIUM_EXTRA`, azure_units.py:16).

#### `app/azure_findings.py`
- **Purpose:** Shasta Azure `Finding` → unified `{entities,edges,findings}`.
- **Key symbols:** `subscription_entity()` (azure_findings.py:20) emits a
  top-level `azure_subscription` entity; `convert_azure_findings()`
  (azure_findings.py:35) parses each finding's `resource_id` (via
  `parse_azure_id`), emits a subject entity + `contains` edge from the
  subscription, drops `not_assessed`/`not_applicable`. `_to_emission()`
  (azure_findings.py:74) maps the 7 standard Shasta framework attrs +
  `merge_framework_map`, rewrites domain `ai_governance` → `ai`.

#### `app/ai_pass.py`
- **Purpose:** Azure cloud-AI discovery + checks → unified emissions.
  Azure analog of `shasta_runner/app/ai_pass.py`.
- **Key symbols:** `discovery_to_entities()` (ai_pass.py:50) maps Shasta
  AI discovery (`azure_openai.accounts` → `azure_openai_deployment`,
  `azure_ml.workspaces` → `azure_ml_workspace`,
  `cognitive_services.accounts` → `cognitive_service`, skipping
  kind=`openai` dupes); `ai_findings_to_emissions()` (ai_pass.py:115)
  pulls AI-framework control IDs (nist_ai_rmf, iso_42001, eu_ai_act,
  owasp_llm_top10, owasp_agentic, nist_ai_600_1, mitre_atlas) from
  `finding.details`; `run_ai_pass()` (ai_pass.py:179) lazily imports
  Shasta and orchestrates. Returns `EntityEmission`/`EdgeEmission`/
  `FindingEmission` directly (not Shasta Findings) — which is why "ai"
  bypasses `convert_azure_findings`.

#### `app/azure_id_to_entity.py`
- **Purpose:** parse an ARM resource ID → entity shape. Analog of
  `arn_to_entity.parse_arn`.
- **Key symbol:** `parse_azure_id(resource_id)`
  (azure_id_to_entity.py:37). `_KIND_MAP` (azure_id_to_entity.py:23)
  covers 10 ARM (namespace, type) pairs → kinds (storage account, VM,
  disk, vnet, NSG, public IP, key vault, SQL server, postgres, app
  service). Unknown type → `None` (finding kept, no entity).

### GCP runner — `platform/lambda/shasta_runner_gcp/`

Structurally a near-clone of the Azure runner. Differences only:

#### `app/main.py`
- **Entry point / handler:** `handler(event, context)` (gcp main.py:94).
- **Key symbols:** `GCP_MODULES` (main.py:71) — 7 modules: iam, storage,
  networking, encryption, compute, logging, cloud_run. **No AI pass** is
  wired (gcp_units.py:10 notes Deep's "+ AI pass" is deferred).
  - **Org-mode branch** (main.py:148): if `mode == "org"`,
    `enumerate_projects(base_client)` lists every visible project, writes
    them to `cloud_connections.scope.projects`, and (if no subset passed)
    scans all of them. Project mode = single onboarded project.
  - `_record_projects()` (main.py:280) — best-effort `scope.projects`
    write.
- **Credentials handling (main.py:113-142):** resolves AWS Fargate task
  creds via `boto3.Session().get_credentials()`, exports them to env
  (`export_aws_credentials_to_env`), then `google.auth.aws.Credentials.
  from_info(...)` builds the WIF external-account credential.

#### `app/gcp_credential.py`
- **Key symbols:** `export_aws_credentials_to_env()` (gcp_credential.py:18)
  — Fargate serves creds via the container endpoint, not env/IMDS, so
  boto3 resolves them and we export `AWS_ACCESS_KEY_ID`/etc. for
  google-auth to sign the `GetCallerIdentity` subject token.
  `build_external_account_info()` (gcp_credential.py:43) builds the WIF
  `external_account` dict (audience = WIF pool/provider; impersonation
  URL for the reader SA). **No private key on disk.** Note: the exported
  AWS creds are a point-in-time snapshot — a scan outliving the cred TTL
  would fail (fine for Quick/Medium; flagged for long Deep scans).

#### `app/project_discovery.py`
- **Key symbols:** `discover_projects()` (project_discovery.py:25) — same
  probe contract as Azure's `discover_subscriptions`.
  `enumerate_projects(client)` (project_discovery.py:56) — org-mode-only,
  `{project_id: display_name}` from `client.list_projects()`.

#### `app/gcp_units.py`
- **Key symbol:** `modules_for_tier()` (gcp_units.py:23). Quick =
  `iam,storage` / `networking,encryption,compute`; Medium & Deep =
  all 7, single phase. **Deep === Medium module-wise** (gcp_units.py:31).

#### `app/gcp_findings.py`
- Direct analog of `azure_findings.py`. `project_entity()`
  (gcp_findings.py:19) emits `gcp_project`; `convert_gcp_findings()`
  (gcp_findings.py:34) emits subject entity + `contains` edge from the
  project. `_to_emission()` (gcp_findings.py:73) is **byte-for-byte the
  same logic** as Azure's.

#### `app/gcp_id_to_entity.py`
- `parse_gcp_id()` (gcp_id_to_entity.py:34) — handles selfLink URLs +
  full resource names; tokenises path, finds last known collection token.
  `_KIND_MAP` (gcp_id_to_entity.py:21) = 9 collections (instances,
  buckets, networks, subnetworks, firewalls, clusters, serviceAccounts,
  keyRings, services). `services` is disambiguated to Cloud Run only when
  `run.googleapis.com` is in the path. **Module docstring warns the exact
  resource_id strings should be confirmed against a live scan (Task 11).**

### Entra runner — `platform/lambda/shasta_runner_entra/`

**Architecturally distinct from Azure/GCP** — no ScanUnit pipeline, no
unified_writer, no broadcast.

#### `app/main.py`
- **Entry point / handler:** `handler(event, context)` (entra main.py:43).
  Event: `scan_id`, `tenant_id`, `conn_id`, `entra_tenant_id`.
- **Auth (main.py:55-66):** loads our app reg's `client_id`/`client_secret`
  from Secrets Manager (`ENTRA_SCANNER_SECRET_NAME`,
  `ciso-copilot/entra-scanner-creds`), sets `AZURE_*` env with
  `AZURE_TENANT_ID = entra_tenant_id` (the **customer's** tenant) BEFORE
  importing Shasta, so `DefaultAzureCredential` issues a customer-scoped
  Graph token. **No subscription scope** — `validate_credentials()` is
  deliberately skipped (it expects ARM access; this is Graph-only).
- **What it enumerates:** `shasta_entra.run_all_azure_entra_checks(client)`
  (main.py:69) — Conditional Access / MFA / guest / privileged-role / etc.
  Entra posture checks. Plus the **AI sign-in pass** (below).
- **Key symbols:**
  - `_FINDING_INSERT_SQL` (main.py:134) — inline upsert into `findings`
    on the natural key `(tenant_id, conn_id, check_id,
    COALESCE(resource_arn,''), COALESCE(region,''))`. **Own SQL — does
    NOT call unified_writer.**
  - `_insert_findings()` (main.py:225) /
    `_insert_finding_param_lists()` (main.py:248) — batch
    (`batch_execute_statement`, batch size 25) with a framework-registry
    pass applied to the param-list shape.
  - `_enrich_param_lists_with_registry()` (main.py:164) — runs the same
    `framework_registry.apply` engine, but against JSON-encoded INSERT
    param dicts with empty `entity_index` (Entra findings have no
    `subject_entity_id`; `ai_touching` can only fire via
    `evidence_packet.is_ai='true'`).
  - `_update_connection_premium_flag()` (main.py:307) — sticky
    `scope.signin_premium_required` flag for Entra-Free-tier 403s.
  - `_fire_personal_tier_pushes()` (main.py:354) — APNs push for new
    `ai_signin_personal_tier` findings (uses `_shared.push`). This is the
    only runner that fires device pushes.

#### `app/ai_signin_pass.py`
- **Purpose:** Shadow-AI sign-in detection from Microsoft Graph audit logs.
- **Key symbols:** `load_catalog()` (ai_signin_pass.py:43) reads
  `ai_saas_catalog.json`; `match_app()` (ai_signin_pass.py:49) matches a
  sign-in event by appId or display-name substring + infers tier
  (personal/corp/unknown); `signin_to_params()` (ai_signin_pass.py:92)
  builds the INSERT param-list (corp tier → pass/low; personal/unknown →
  fail; `evidence_packet.is_ai='true'` + `entra_upn`);
  `run_ai_signin_pass()` (ai_signin_pass.py:160) pages
  `/auditLogs/signIns`, returns `(param_lists, premium_required)`.
  `_fetch_signins()` (ai_signin_pass.py:204) sets `premium_required` only
  on the licensing-403 error code
  `Authentication_RequestFromNonPremiumTenantOrB2CTenant`. Graph SDK
  imported lazily; single-page fetch (`top=1000`) — multi-page paging is
  a flagged follow-on.

### Shared (build-time-copied) modules

These appear in `azure/app/` and `gcp/app/` but are **NOT git-tracked
there** — `build.sh` copies them in from the source-of-truth siblings
(`scanner_core/`, `ai_scanner/`, `_shared/`). They are byte-identical
across runners (verified `diff -q`):

- `scan_pipeline.py` ← `scanner_core/scan_pipeline.py` — `ScanUnit`,
  `ConcurrencyLimiter`, `run_units` (thread-pool fan-out, per-service
  semaphores, per-unit failure isolation). Cloud-agnostic.
- `scan_state.py` ← `scanner_core/scan_state.py` — `update_scan` /
  `record_scan_scope` against the `scans` table. Cloud-agnostic by design
  (the `scope` dict is opaque: region-keyed for AWS, subscription-keyed
  for Azure, project-keyed for GCP).
- `unified_writer.py` ← `ai_scanner/unified_writer.py` — `commit_scan`
  (transactional entity/edge/finding upsert + framework-registry pass +
  **post-commit broadcast fan-out**) / `mark_scan_failed`. Used by Azure +
  GCP. NOT used by Entra.
- `framework_registry.py` + `ai_framework_registry.json` ←
  `scanner_core/` — CME-v2 two-stage normalize→augment crosswalk engine.
  Copied into all three runners.
- `detectors/base.py` ← `ai_scanner/detectors/base.py` — `EntityEmission`
  / `EdgeEmission` / `FindingEmission` / `UnitResults` dataclasses. Azure
  + GCP only.
- `_shared/broadcast_fanout.py` ← `_shared/` — `publish_if_critical`
  CRITICAL→SQS broadcast. Staged into Azure + GCP `app/`; for Entra it is
  COPYed to `${LAMBDA_TASK_ROOT}/_shared/` but `main.py` never calls it.

`framework_map.py` is the exception — it IS git-tracked in each runner
(`shasta_runner/`, `_azure/`, `_gcp/`, `_entra/`), i.e. four hand-mirrored
copies. Its docstring (framework_map.py:1-6) claims a
`sync_framework_map.py` mirrors it — **that script does not exist in the
repo** (grep: 0 hits). See Dead code.

## Data & contracts touched

- **DB tables:**
  - `scans` (read/write) — columns `status`, `phase`, `stats` (jsonb),
    `scope` (jsonb), `error`, `finished_at`, keyed by `scan_id`.
  - `ai_scans` (write, via unified_writer) — `status`, `completed_at`,
    `assets_discovered_count`, `relationships_discovered_count`,
    `findings_generated_count`, `scanner_version`, `error_message`, keyed
    by `id`.
  - `entities` — upsert on `(tenant_id, kind, natural_key)`, PK `id`.
  - `edges` — upsert on `(source_entity_id, target_entity_id, kind)`.
  - `findings` — upsert on `(tenant_id, conn_id, check_id,
    COALESCE(resource_arn,''), COALESCE(region,''))`, PK `finding_id`.
    Entra uses its own copy of this SQL.
  - `cloud_connections.scope` (jsonb) — Azure writes `subscription_names`;
    GCP writes `projects`; Entra writes/clears `signin_premium_required`.
- **Env vars:** `DB_CLUSTER_ARN`, `DB_SECRET_ARN`, `DB_NAME` (all three);
  Azure/GCP additionally `AUTONOMOUS_BROADCAST_QUEUE_URL` (via RunTask
  override); Entra additionally `ENTRA_SCANNER_SECRET_NAME`,
  `APNS_PLATFORM_APP_ARN`, and `AUTONOMOUS_BROADCAST_QUEUE_URL` (set but
  unused). Azure/GCP per-scan params arrive as RunTask container-override
  env (see `run.py`).
- **AWS resources:** ECS task defs `azureScanTaskDef`; task role
  `gcpScannerRole`; ECR repos `shasta-runner-{azure,gcp,entra}`; Secrets
  Manager `ciso-copilot/entra-scanner-creds`; the autonomous broadcast SQS
  queue.
- **External APIs:** Azure Resource Manager + Microsoft Graph (Azure,
  Entra); Google Cloud APIs via WIF (GCP).

## Broadcast plumbing status (verified)

The recent fix **`eba9574`** ("fix(broadcast): inject
AUTONOMOUS_BROADCAST_QUEUE_URL into Azure + GCP RunTask overrides",
2026-06-03) is present and correct:

- Lambda scanners (`shastaRunner`, `shastaRunnerEntra`, `aiScanner`) get
  `AUTONOMOUS_BROADCAST_QUEUE_URL` via CDK `addEnvironment`
  (scan-stack.ts:410-411) plus the SQS grant.
- The Azure + GCP **Fargate** task roles get only the IAM grant
  (scan-stack.ts:416-417). CDK `ContainerDefinition` has no
  post-construction env-mutation API, so the queue URL is injected as a
  `containerOverrides[].environment` entry at `ecs.run_task` dispatch
  time. Verified at all four RunTask sites:
  - `connections_list/main.py:289` (Azure rescan) + `:388` (GCP rescan)
  - `onboarding_azure_complete/main.py:197`
  - `onboarding_gcp_complete/main.py:211`
  - Each gated `... if AUTONOMOUS_BROADCAST_QUEUE_URL else []` (backward
    compatible: empty env → no override → `broadcast_fanout`
    short-circuits at broadcast_fanout.py:61).
- The actual fan-out runs inside `unified_writer.commit_scan`
  **post-commit** (unified_writer.py:143-159) — `publish_if_critical` per
  finding, after `commit_transaction` succeeds (the in-tx-broadcast race
  is documented at unified_writer.py:84-89). So Azure + GCP fan out
  CRITICAL findings to Slack correctly via the shared writer.
- **Entra does NOT fan out** — it writes findings with its own
  `batch_execute_statement` (entra main.py:239) and never imports
  `broadcast_fanout`. Its `AUTONOMOUS_BROADCAST_QUEUE_URL` env + SQS grant
  are dead wiring (see Dead code). Entra instead fires **APNs pushes**
  (its own broadcast channel).

## CLAUDE.md "Don't lean on Azure Sentinel" — what the Azure scanner queries

Verified: the Azure runner does **not** touch Sentinel. Its modules
(`AZURE_MODULES`, main.py:69) are Shasta's `run_all_azure_*_checks` for
iam, governance, storage, networking, compute, encryption, databases,
appservice, monitoring, backup, diagnostic_settings, private_endpoints.
The `framework_map.py` Azure check IDs confirm the data sources align
with the guidance — Activity Log (`azure-activity-log*`,
framework_map.py:169-174), Policy (`azure-policy-compliance`,
`azure-security-initiative`), Resource-Graph-style resource posture
(storage/sql/network/keyvault checks), and **Defender-if-on**
(`azure-defender-enabled`, `azure-defender-per-plan`,
framework_map.py:205-206). No Sentinel / Log Analytics ingestion path
exists in this runner. (The Shasta module internals are out of scope, but
the invoked check set and mapped controls are consistent with
Activity Log + Policy + Resource Graph + Defender-if-on.)

## Dead code (suspected)

1. **`framework_map.py` docstring reference to `sync_framework_map.py`**
   (framework_map.py:4) — **High confidence the script is missing.**
   `grep -rn "sync_framework_map" platform/` → 0 hits beyond the four
   docstrings. The four `framework_map.py` copies are git-tracked and must
   be hand-synced; the "never hand-edit those copies / sync mirrors this"
   instruction points at a script that doesn't exist. Not dead code per se,
   but a **broken process invariant** a spec author will trip over.

2. **Entra runner's `AUTONOMOUS_BROADCAST_QUEUE_URL` env + SQS grant**
   (scan-stack.ts:410-412) — **High confidence unused by Entra.**
   `grep -rn "broadcast_fanout" shasta_runner_entra/` → 0 hits; entra
   `main.py` writes findings via `batch_execute_statement` and never
   imports the fan-out. The env var and `q.grantSendMessages(fn)` are
   inert for the Entra Lambda (harmless, but a spec author wiring CRITICAL
   broadcast for Entra should know it is NOT plumbed today — Entra
   findings never reach Slack, only APNs).

3. **`azure_units.ALL_MODULES` / `gcp_units.ALL_MODULES`**
   (azure_units.py:19, gcp_units.py:20) — **Med confidence "test-only".**
   `grep -rn ALL_MODULES` shows the engine path uses `modules_for_tier`;
   `ALL_MODULES` is referenced only by `tests/test_*_units.py`. Not truly
   dead (it's an exported test anchor), but no runtime caller. Leave it —
   removing it would break the tier-coverage tests.

(No truly orphaned functions found in the runner source. `subscription_entity`,
`project_entity`, `enumerate_projects`, `_record_*`, `_ai_unit`,
`ai_signin_pass.*` are all wired.)

## Optimization opportunities

1. **Cross-runner duplication of `_to_emission` + `convert_*_findings` +
   `*_id_to_entity` + the 3-stage `handler`/`_build_units`/`_module_unit`/
   `_absorb`/`_record_*` scaffold.** `azure_findings._to_emission` and
   `gcp_findings._to_emission` are line-for-line identical; the Azure and
   GCP `main.py` `handler`s differ only in nouns (subscription↔project),
   the credential block, and Azure's `_ai_unit` branch / GCP's org-mode
   branch. This is ~250 lines duplicated per runner. A shared
   `scanner_core/cloud_runner.py` (parameterised by a small per-cloud
   adapter: top-level entity kind, id-parser, module map, credential
   setup, AI-pass hook) would collapse three near-clones into one engine +
   three thin adapters — the same move the docstrings already gesture at
   ("so it can be shared by the Azure / GCP scanners later", scan_pipeline.py:9).
   Cost: a build-time refactor touching all three Dockerfiles' COPY sets.

2. **`framework_map.py` four-copy hand-sync.** Make `framework_map.py` a
   build-time copy from a single source-of-truth (the same pattern already
   used for `framework_registry.py` / `unified_writer.py` / `scan_state.py`
   in `build.sh`), or actually add the referenced `sync_framework_map.py`.
   Today a control-mapping edit must be applied to four files by hand; drift
   is a matter of when, not if. Concrete: move the master to
   `scanner_core/framework_map.py`, add `cp ../scanner_core/framework_map.py
   app/framework_map.py` to each `build.sh`, gitignore the runtime copies.

3. **Entra AI sign-in single-page fetch** (ai_signin_pass.py:230-240). Only
   the first `top=1000` page of `/auditLogs/signIns` is read; high-volume
   tenants silently lose sign-ins beyond page 1. The code already flags this
   ("Production-quality paging across many pages is a follow-on"). Add the
   SDK's async page iterator (or `@odata.nextLink` follow) before relying on
   Entra shadow-AI coverage for large tenants.

(Bonus, lower priority: GCP exports point-in-time AWS creds for WIF
signing — gcp_credential.py:34-35 flags that a Deep scan outliving the
cred TTL would fail. Not a bug today; revisit if Deep wall-clock grows.)

## Gotchas / non-obvious behavior

- **Three near-identical runners, but Entra is structurally different.**
  Azure + GCP share the `scan_pipeline` / `scan_state` / `unified_writer`
  stack and the ScanUnit fan-out. Entra is a single-pass Lambda with its
  own inline `findings` upsert SQL, its own registry-application path
  (param-list shaped, empty `entity_index`), no entities/edges, no
  broadcast. Don't assume "the scanners" behave uniformly.
- **Shared modules in `*/app/` are build-time copies, not source.** Editing
  `shasta_runner_azure/app/unified_writer.py` is editing a gitignored
  artifact — change `ai_scanner/unified_writer.py` (and
  `scanner_core/*`, `_shared/*`) instead, then rebuild. `git ls-files`
  confirms these are untracked in the runner dirs.
- **`framework_map.py` is the one shared-looking file that IS tracked per
  runner** — four manual copies, no working sync script. Edit all four.
- **Azure runs an AI pass (Medium+Deep); GCP does not.** "ai" is in Azure's
  `_MEDIUM_EXTRA` and wired via `_ai_unit`; GCP's Deep tier is
  module-identical to Medium with the AI pass explicitly deferred
  (gcp_units.py:10).
- **The AI pass bypasses `convert_*_findings`.** `run_ai_pass` already
  returns `EntityEmission`/`EdgeEmission`/`FindingEmission`, so `_ai_unit`
  feeds its dict straight into the accumulators — a normal module unit goes
  through `convert_azure_findings` first.
- **Per-unit fresh client is load-bearing.** Both Azure and GCP build a new
  SDK client per ScanUnit inside `run` and call `validate_credentials()`
  there — `for_subscription`/`for_project` do NOT validate, and they exist
  precisely so concurrent threads don't share mutable SDK state.
- **Credential injection happens via `os.environ` before Shasta import.**
  Azure SP creds (azure_credential.py) and Entra app creds (entra
  main.py:56-58) are written to `AZURE_*` env so `DefaultAzureCredential`
  resolves them. Entra sets `AZURE_TENANT_ID` to the **customer** tenant,
  not ours — the token is customer-scoped.
- **GCP WIF needs AWS creds in env.** Fargate serves task creds via the
  container endpoint (not env/IMDS), which google-auth's AWS source can't
  read — so the runner resolves them with boto3 and exports them
  (gcp_credential.py:18) before building the WIF credential. A point-in-time
  snapshot, not refreshed.
- **GCP org mode self-enumerates.** If `mode == "org"` and no `project_ids`
  subset is passed, the scanner lists every visible project, writes them to
  `scope.projects`, and scans all of them — the webhook stores an empty list
  intentionally (onboarding_gcp_complete docstring).
- **Findings upsert natural key has no `kind` and uses `''` coalesced
  resource/region.** `(tenant_id, conn_id, check_id, COALESCE(resource_arn,''),
  COALESCE(region,''))`. The broadcast receiver looks up by the RETURNING
  `finding_id`; a mismatched id silently drops the Slack card — which is why
  both writers return the persisted id, never the client UUID
  (unified_writer.py:317-382).
- **Entra fires the only device pushes.** `_fire_personal_tier_pushes`
  (entra main.py:354) pushes new `ai_signin_personal_tier` findings via APNs
  — non-fatal, post-write. No other runner pushes.
- **`scan_state` vs `unified_writer` write to different scan tables.**
  `scan_state.update_scan` targets `scans` (status/phase/stats/scope);
  `unified_writer._update_scan` targets `ai_scans` (counts/scanner_version).
  Both fire per scan in Azure/GCP. Entra only touches `scans`.
