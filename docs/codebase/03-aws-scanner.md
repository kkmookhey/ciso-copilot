## Overview

`shasta_runner` is the **AWS posture scanner**. One Docker image (`Dockerfile`)
runs in **two modes from the same code**: as a Lambda
(`ShastaRunner` DockerImageFunction, CMD `main.handler`, `platform/lib/scan-stack.ts:76`)
and as the `ciso-copilot-aws-scan` **ECS Fargate task** whose entrypoint is
`python run.py` (`scan-stack.ts:94-137`). The Fargate path is the live one: the
onboarding / rescan Lambdas (`onboarding_aws_complete`, `connections_list`) fire
`ecs:RunTask` with scan params as container env-var overrides
(`run.py:5`, `connections_list/main.py:44,230`). `run.py` maps those env vars into
the event dict `main.handler` expects (`run.py:27-45`).

End-to-end flow (`main.py:14-25` docstring + `handler` at `main.py:171`):
1. `UPDATE scans SET status='running'` (`scan_state.update_scan`).
2. STS `AssumeRole` into the customer account → **auto-refreshing** credentials
   (`assumed_role.build_refreshable_credentials`) so a long multi-region scan
   never expires mid-flight.
3. **Region discovery** (`region_discovery.discover_regions`): enumerate enabled
   regions, probe each in parallel, classify `active`/`default_only`/`empty`/`unknown`.
4. **Scan plan** (`scan_policy.build_scan_plan`): the `(scan_tier × region_state) → depth`
   matrix decides which modules run where.
5. **Build + run ScanUnits** through the shared thread-pool pipeline
   (`scan_pipeline.run_units`): global enums (IAM, S3), global Shasta modules,
   AI pass, per-region enums (compute, network), coverage engine, regional Shasta.
6. Shasta `Finding` objects → `FindingEmission`; ARNs → subject entities + `contains` edges
   (`main.convert_shasta_findings`).
7. **Transactional DB write** via `unified_writer.commit_scan` (entities → edges →
   findings, framework registry applied per finding, post-commit CRITICAL broadcast).
8. `UPDATE scans` to `completed` / `partial` + stats + scope coverage map.

Per-unit failures are isolated (`scan_pipeline.run_units`), so one bad module
yields a `partial` scan, never a dead one. **Quick** tier commits in two phases
(global early-commit "First Signal", then per-region "Crown Jewel"); **medium/deep**
commit once (`main.py:228-248`).

## Framework registry — VERIFIED counts

`ai_framework_registry.json` (identical in `scanner_core/` and the runtime copy in
`app/`) has **two top-level keys: `frameworks` and `rules`**.

- **AI-family framework packs: 8** (all `family: "ai"`):
  `nist_ai_rmf`, `nist_ai_600_1`, `iso_42001`, `eu_ai_act`, `owasp_llm_top10`,
  `owasp_agentic`, `mitre_atlas`, `soc2_ai`.
- **Mapping rules: 13** (`rules[]`): `ai_signin_personal_tier_controls`,
  `ai_signin_corp_tier_controls`, `ai_signin_unknown_tier_controls`, and 10
  `baseline_*` rules (bedrock content-filter/vpc-endpoint, cloudtrail-ai-events,
  s3-training-data-versioned, and 6 sagemaker baselines).
- Each framework pack carries metadata (`name`, `source`, `family`, `source_url`,
  `version`, `canonical_format`) and optional `rewrite_rules`. **5 of 8** packs
  have `rewrite_rules` (65 rewrite entries total) used by `_normalize_stage`.

**CLAUDE.md's "8 AI-family packs + 13 mapping rules" claim is CORRECT** (verified
by counting the JSON, 2026-06-05).

How findings get mapped (the full chain, in order):
1. Shasta native framework attrs → `frameworks` (soc2, cis_*, mcsb, iso27001, hipaa) — `main.py:478`.
2. `framework_map.merge_framework_map` adds FedRAMP + PCI DSS by `check_id` — `main.py:488`.
3. `ai_pass` pulls NIST AI RMF / ISO 42001 / EU AI Act / OWASP LLM+Agentic / NIST 600-1 / MITRE ATLAS out of `finding.details` — `ai_pass.py:164`.
4. `unified_writer.commit_scan` calls `framework_registry.apply` per finding — normalize (rewrite control IDs to canonical published format) then augment (registry `rules`) — `unified_writer.py:104-112`.

Note: the JSON registry's `rules: [...]` is non-empty (13 rules), but the
`ai_touching` selector still never gates because stub entities aren't backfilled
into `entity_index` and no shipped rule depends on it (`unified_writer.py:60-63`).

## Data & contracts touched (unified_writer.py)

**`unified_writer.commit_scan(ctx, entities=, edges=, findings=)` is the single
transactional DB-write boundary.** One `begin_transaction`; rollback on any error.
Order: entities → edges → findings → `ai_scans` row update → commit → post-commit broadcast.

Tables written (all via RDS Data API):
- **`entities`** — `_upsert_entity` (`:178`). `INSERT … ON CONFLICT (tenant_id, kind, natural_key) DO UPDATE SET last_seen_at=NOW(), attributes (minus `_stub`), evidence_packet, display_name … RETURNING id::text`. **Always uses the RETURNING id** (existing row on conflict), never the client UUID. Columns: `id, tenant_id, kind, natural_key, display_name, domain, attributes(jsonb), evidence_packet(jsonb), detector_id, detector_version, scan_id`.
- **`edges`** — `_upsert_edge` (`:283`). `ON CONFLICT (source_entity_id, target_entity_id, kind) DO UPDATE`. Targets/sources resolved by `(kind, natural_key)`; a reference to an entity not in this scan → `_resolve_or_stub` (`:258`) creates a stub `{_stub: true}` entity (`_domain_for` infers domain from kind prefix, `:273`). Columns: `id, tenant_id, source_entity_id, target_entity_id, kind, attributes, evidence_packet, detector_id, detector_version, scan_id`.
- **`findings`** — `_insert_finding` (`:317`). `ON CONFLICT (tenant_id, conn_id, check_id, COALESCE(resource_arn,''), COALESCE(region,'')) DO UPDATE` — re-scans refresh in place (preserves `first_seen`, refreshes `last_seen`, clears `resolved_at`). **Returns the persisted `finding_id`** for the broadcast. Columns: `finding_id, tenant_id, conn_id, scan_id, check_id, title, description, severity, status, resource_arn, resource_type, region, domain, frameworks(jsonb), remediation, first_seen, last_seen, evidence_packet(jsonb), subject_entity_id`. (`conn_id` comes from `ctx.connection_id`; `subject_type` defaults to `'ai_module'` if null, `:363`.)
- **`ai_scans`** — `_update_scan` (`:385`): `status, completed_at, assets_discovered_count, relationships_discovered_count, findings_generated_count, scanner_version`. `mark_scan_failed` (`:162`) sets `status='failed', completed_at, error_message`.

Framework-registry pass runs **inside** the finding loop (`apply_registry`, `:105`);
`FindingEmission` is frozen so the result is `dataclasses.replace`d (`:109`).
Aggregated counters logged as `registry_apply_summary` (`:124`).

**Broadcast-after-commit (fix e14b1f5):** broadcast tuples are accumulated during
the in-tx insert loop (`pending_broadcasts`, `:89,120`) and fired ONLY **after**
`commit_transaction` succeeds (`:146-159`). Rationale in the code comment
(`:84-89`): the subscriber Lambda's `SELECT WHERE finding_id=:fid` triggers within
ms and would run before the scanner commits, silently dropping the Slack card.
Post-commit guarantees row visibility. Each `publish_if_critical` is wrapped
defensively so one bad message can't tank the rest.

`scans` table (separate from `ai_scans`): `scan_state.update_scan` writes
`status, phase, stats(jsonb), error, finished_at`; `record_scan_scope` writes
`scope(jsonb)`. **`scans` and `ai_scans` are different tables** — the scanner
updates `scans` for status/phase and `ai_scans` for asset/finding counts.

Env vars: `DB_CLUSTER_ARN, DB_SECRET_ARN, DB_NAME` (writer + scan_state),
`AUTONOMOUS_BROADCAST_QUEUE_URL` (broadcast_fanout, optional).

## Optimization opportunities

- `enumerate_storage.py:37,68` — **N+1 `get_bucket_location` call per bucket** (serial, inside the bucket loop). For accounts with many buckets this serializes on a per-bucket API call. Could batch/parallelize or skip when region isn't needed. Low priority (each call is cheap + best-effort).
- `ai_pass.discover_bedrock_and_ai_lambdas:227-259` — **serial region loop** for Bedrock guardrails + Lambda AI-var discovery. Every other multi-region path (region_discovery, the handler's unit pool) is parallelized; this one isn't. A region with many functions paginated serially extends the single `ai` ScanUnit's wall-clock. Consider a `ThreadPoolExecutor` over `regions` like `region_discovery` does.
- `unified_writer.py` — **one RDS Data API round-trip per entity / edge / finding** (`_upsert_*` each call `execute_statement` individually). For a large scan this is hundreds of synchronous calls inside one transaction. `batchExecuteStatement` (RDS Data API) could collapse the per-row inserts. Concrete but non-trivial (the per-row `RETURNING id` dependency complicates batching for entities/findings).

## Gotchas / non-obvious behavior

- **Two tables, not one:** the scanner writes status/phase/scope to **`scans`** but asset/relationship/finding counts to **`ai_scans`**. A spec touching scan lifecycle must update the right one.
- **`scanner_core/` is the source of truth; `app/` copies are generated.** Edit `framework_registry.py` / `scan_pipeline.py` / `scan_state.py` / `ai_framework_registry.json` in `platform/lambda/scanner_core/`, and `unified_writer.py` / `detectors/base.py` in `platform/lambda/ai_scanner/`, and `broadcast_fanout.py` in `platform/lambda/_shared/`. `build.sh` re-copies them; the `app/` copies are `.gitignore`d.
- **AI compliance frameworks already exist.** NIST AI RMF, NIST AI 600-1, ISO 42001, EU AI Act, OWASP LLM Top 10, OWASP Agentic, MITRE ATLAS, SOC2-AI are all live (8 packs in `ai_framework_registry.json` + 13 mapping rules). Don't propose adding them as "new" — they ship today. (This is the exact mistake the CLAUDE.md "verify before claiming new" rule was written to prevent.)
- **FedRAMP + PCI DSS mappings live in `framework_map.py`** (~320 check IDs across AWS/Azure/GCP/GitHub), a *separate* mechanism from the JSON registry. Two crosswalk systems coexist by design.
- **Broadcast must stay post-commit** (`commit_scan:146`). Moving `publish_if_critical` back inside the transaction reintroduces the silent-drop race fixed in e14b1f5.
- **Coverage engine covers only 3 services** (sqs, secretsmanager, ecr — `coverage/registry.py:19-30`). It's a real, wired pass but a thin one; most posture comes from Shasta's 16 modules.
- **EC2→role edges use the instance-profile ARN, not the role ARN** (`enumerate_compute.py:85-105`) — the writer stubs the target. Don't treat that `aws_iam_role` natural key as a real role ARN.
- **Region anti-blind-spot rule:** any region whose probe errors is classified `unknown` and scanned at full depth (`region_discovery.classify_region:48`, `scan_policy:14`). A scan is never silently narrowed.
- **Quick tier = two commits** (early "first_signal" global commit, then "crown_jewel" per-region) — `main.py:228-242`. A finding can appear after the first commit. Medium/deep = single commit.
- **Findings dedupe on `(tenant_id, conn_id, check_id, resource_arn, region)`** (`_insert_finding:341`). Re-scans update in place; `first_seen` is preserved, `resolved_at` cleared. Changing a check_id orphans its prior findings (they won't flip to resolved).
- **The same image is both the Lambda and the Fargate task.** `main.handler` (Lambda) and `run.py` (Fargate, the live path triggered by `ecs:RunTask`). Don't assume Lambda-only constraints; the live scanner runs on Fargate.
- **`findings.frameworks` is a JSON object** (`{}`), never an array (matches the broader Aurora schema gotcha in CLAUDE.md).
