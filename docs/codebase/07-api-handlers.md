# Read/Query API Lambda Handlers — Codebase Documentation

> Generated by codebase-documentation sweep 2026-06-05. Purpose: give a
> future spec author a grep-able answer to "does this API endpoint already
> exist?" before they propose building it. Cite `file:line` everywhere.
> Scope: the read/query API Lambda handlers under `platform/lambda/`.

## Overview

These Lambdas are the customer-facing read/query (+ light CRUD) surface of
Shasta. They sit behind a single API Gateway `RestApi` with stage name `v1`
(`platform/lib/api-stack.ts:454-457`), Cognito-authorized (`authedOpts`)
except `GET /public/trust/{slug}`. Each Lambda is a flat-zipped Python
handler (`lambda.Code.fromAsset`, so imports are bare `from X import Y`, no
package paths). All DB access is via the Aurora Data API
(`rds_data.execute_statement`) against database `ciso_copilot`. Triggers are
HTTP routes only — except `scan_reaper`, which is a scheduled stuck-scan
sweep (see its section; **source is missing from the working tree**).

Every authed handler resolves `tenant_id` the same way: extract the Cognito
subject, then `SELECT tenant_id FROM users WHERE sso_subject = :s`. The
canonical subject-extraction (`identities[0].userId` first, then `sub`) is
duplicated into each Lambda because each is a separate zip.

## Tenant-scoping correctness (the Cognito gotcha)

All authed handlers MUST resolve `tenant_id` via `users.sso_subject`, and
the subject MUST be taken from `identities[0].userId` first, falling back to
`sub` — because for federated logins `claims.sub` is the Cognito-pool sub,
NOT the upstream IdP sub stored in `users.sso_subject`. Canonical impl:
`events_list._resolve_tenant_id`, `voice_session._subject_from_claims`.

**All in-scope handlers get this RIGHT.** Each carries the identical
`identities[0].userId → sub` pattern:

- `risks/main.py:229-251`
- `findings_list/main.py:162-184`
- `findings_rollup/main.py:153-175`
- `findings_summary/main.py:95-117`
- `entities_api/helpers.py:32-59` (`resolve_tenant_id` / `_subject_from_claims`)
- `compliance_summary/main.py:110-132`
- `scans_status/main.py:25-56`
- `connections_list/main.py:612-637`
- `questionnaires/main.py:443-465`
- `policies/main.py:403-425`
- `me/main.py:81-96` (`_resolve_subject`)
- `trust/main.py:282-304`

No handler reaches for `claims.get("sub")` directly without the `identities`
fallback. **Zero risky handlers found in this subsystem.**

Note one tenant-isolation subtlety (not a bug, but worth knowing): a few
secondary queries omit the `tenant_id` filter because they key off an
already-tenant-verified id:
- `scans_status/main.py:83-86` counts `findings` by `scan_id` only (the
  parent `scans` row was already tenant-scoped at line 70).
- `questionnaires/main.py:274-280` (`_get` items) and `:391-429` (`_patch_item`
  UPDATE) filter by `questionnaire_id` after a tenant-ownership check on the
  parent `questionnaires` row (lines 259-271 / 393-403). Correct but relies on
  the prior check.
- `entities_api` `_entity_graph` edge-fetch (`main.py:467-479`) and the node
  CTE both filter `tenant_id`, so the graph cannot cross tenants.

## Dead code (suspected)

1. **`questionnaires/main.py:52` (`_from_excel` dispatch) and `:54`
   (`_suggest_item` dispatch) — UNREACHABLE. Confidence: HIGH.**
   The handler checks `if method == "POST": return _create(...)` at line 50
   *before* the `from-excel` check (line 52) and the item-suggest check

     `qid and iid` POSTs before the bare POST). Grep evidence: the only
     callers of `_from_excel`/`_suggest_item` are their own `def` + the
     shadowed dispatch lines.

## scan_reaper — scheduled stuck-scan sweep

Committed on `origin/main` (commit `8a9db88`, PR #42); deployed as
`ciso-copilot-scan-reaper`. Behaviour below was first reverse-engineered from the
working-tree `.pyc` and matches the committed source:

- **Trigger:** EventBridge schedule (the 10-minute sweep referenced in
  MEMORY.md; the rate is not in the current CDK tree). Docstring:
  "Scheduled stuck-scan reaper."
- **Why it exists (GuardDuty gotcha):** onboarding/rescan Lambdas only mark
  a scan `failed` when ECS `RunTask` throws synchronously. If a Fargate task
  dies later (e.g. the GuardDuty Fargate agent sidecar 403s and kills the
  task — see MEMORY.md `project_scan_guardduty_gotcha`), the `scans` row is
  stuck at `queued`/`running` forever and the UI hangs.
- **What it does:**
  1. `_stale_scan_ids`: `SELECT scan_id::text FROM scans WHERE status IN
     ('queued','running') AND started_at < now() - (:grace * interval '1
     minute')` (env `GRACE_MINUTES`).
  2. `_live_scan_ids`: reads `SCAN_ID` env override of every `RUNNING`/
     `PENDING` ECS task on the scan cluster (env `SCAN_CLUSTER_ARN`,
     `desiredStatus`).
  3. `_reap`: for each stale scan NOT backed by a live task,
     `UPDATE scans SET status='failed', finished_at=now(), error =
     COALESCE(error, :err) WHERE scan_id = CAST(:sid AS UUID) AND status IN
     ('queued','running')`. Error text: "reaped: no live scan task after …
     please re-scan".
- **Key safety property:** the live-task check means a legitimately long
  Fargate scan is never reaped regardless of age (the scanner bumps `phase`,
  not a timestamp, so elapsed time alone can't distinguish "slow" from
  "dead"). Connections_list's failure handlers (`scans SET status='failed'`)
  are the synchronous counterpart; the reaper is the async backstop.

## Optimization opportunities

1. **`findings_list/main.py:97-119` — runs TWO queries per request** (page +
   COUNT(*)) under the same filters. Acceptable for pagination but the COUNT
   re-scans `findings` on every page. If `findings` grows large, consider a
   windowed `COUNT(*) OVER()` in the page query to fold them into one round
   trip.
2. **`connections_list._list_connections` (`main.py:79-98`) — LATERAL
   per-connection latest-scan subquery.** Fine for a handful of connections,
   but it's a correlated subquery per row; if a tenant ever has many
   connections this becomes N lookups. Low priority (tenants have ≤4 clouds).
3. **`questionnaires._list` (`main.py:232-242`) — two correlated `(SELECT
   COUNT(*) …)` subqueries per questionnaire row** (total + answered). For a
   tenant with many questionnaires this is 2N item-table scans. Replace with
   a single `LEFT JOIN … GROUP BY` aggregate over `questionnaire_items`.
4. **`compliance_summary` / `trust._aggregate_compliance` — duplicate logic.**
   Both unnest `findings.frameworks` and roll up identical
   passing/failing/score math (`compliance_summary/main.py:46-101` vs
   `trust/main.py:93-127`). A future refactor could share one query, but they
   live in separate zips so it would mean a shared layer. Not urgent.
5. **`entities_api._entity_graph` recursive CTE has no per-node visited cap
   inside the recursion** beyond `walked.depth < :max_depth`; the outer
   `LIMIT :max_nodes` truncates after the walk. On a dense graph the CTE can
   expand widely before truncation. Acceptable at depth≤8 but worth knowing.

## Gotchas / non-obvious behavior

- **Routing-order bug in `questionnaires` (see Dead code #1):** `from-excel`
  and item-level AI-suggest POSTs are shadowed by the bare `POST` handler.
  Anyone "adding" excel upload or AI-suggest should know the code already
  exists (`_from_excel` line 61, `_suggest_item` line 127) — it just needs
  the dispatch reordered, not a rewrite.
- **`findings` has no `kind` column and `status` has no `open` value.**
  Filters use `check_id` as the rule identifier; default status filter is
  `fail`. `frameworks` is a JSONB **object** queried with the `?` operator
  (`findings_list/main.py:74`) and `jsonb_each`/`jsonb_array_elements_text`
  (`compliance_summary/main.py:53-55`).
- **Two distinct scan tables.** `scans` (cloud posture, PK `scan_id`, served
  by `scans_status` + `connections_list`) vs `ai_scans` (repo/AI scans, PK
  `id`, served by `entities_api`). `scans_status` reads `scans`;
  `entities_api` reads `ai_scans`. Don't conflate them.
- **`entities` carry no `connection_id`.** `entities_api._get_entity`
  hard-codes `"connection_id": None` (`main.py:403`); the linkage lives on
  `ai_scans.connection_id`. Containment is via `edges.kind`, not a
  `parent_id` column.
- **`POST /ai/scans` upserts a `github_repo` entity** (`_upsert_repo_entity`,
  `main.py:147-191`) ON CONFLICT (tenant_id, kind, natural_key); natural_key
  is `github.com/{repo_full_name}`.
- **Idempotent creates** in `risks` and `policies` use `ON CONFLICT
  (tenant_id, source_approval_id) DO NOTHING RETURNING …` then re-SELECT the
  winner on conflict — closes the SELECT-then-INSERT TOCTOU window. Reuse
  this pattern for any approval-card-driven create.
- **`trust` has a genuinely public route** (`GET /public/trust/{slug}`,
  registered without `authedOpts`, `api-stack.ts:584-585`). It resolves the
  tenant from the `slug`, NOT from a JWT, and only emits aggregate counts /
  scores per the `show_*` flags. Any new public-facing posture data must go
  through this redaction path.
- **`policies` Lambda has a 5-minute timeout + 1024 MB** (`api-stack.ts:335`)
  because `generate-all` enriches ~8 policies in parallel via Anthropic.
  Other read Lambdas use defaults.
- **`/connections` PATCH validates the picker selection** against the
  connection's discovered `scope.subscriptions` (Azure) or `scope.projects`
  keys (GCP), and refuses an empty selection (`connections_list/main.py:434-447`).
- **`me` returns `{"user": null, "tenant": null}` with HTTP 200** when the
  user row doesn't exist yet (`main.py:62`) — iOS polls this during
  pending-approval; don't change it to a 404.
- **`findings_summary` `by_status` counts ALL findings; `by_severity`/
  `by_cloud` count only `('fail','partial')`** (`main.py:40-42`). The status
  literals are code-controlled (never user input), so the inline `IN (...)`
  is intentional and safe.
- **`scan_reaper` source is on `origin/main`** (commit `8a9db88`, PR #42) and deployed
  (`ciso-copilot-scan-reaper`); it's just not on every feature branch. Earlier drafts of
  this doc wrongly called it uncommitted — corrected 2026-06-08.
