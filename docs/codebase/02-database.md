## Overview

A single Aurora PostgreSQL cluster (`ciso_copilot` database) holds all
multi-tenant state for Shasta. Every business table is tenant-scoped by a
`tenant_id UUID REFERENCES tenants(tenant_id)` — the **only** exception is
`threat_indicators` (global IOC table, intentionally tenant-less; `013:7`).
Access is exclusively through the RDS Data API from Lambda (`rds_data.execute_statement`);
there is no ORM and no `src/repos/` layer in this repo — handlers write raw
parameterized SQL.

The schema grew in three eras: **Phase 0** (`schema.sql`) = tenancy +
identity + audit; **Phase A** (`002`) = the core cloud-posture model
(connections → scans → findings/events); then a series of feature slices
(risks, AI-security, unified entity graph, chat, SOC, threat intel, MCP
connectors). Two parallel modeling attempts coexist: the **legacy
domain-specific** tables (`assets`, `ai_assets`, `ai_relationships`) and the
**unified entity graph** (`entities` + `edges`, migration `005`) that
superseded them. The legacy AI tables are now dead (see Dead code).

**One table is created out-of-band**: `policies` has **no `CREATE TABLE` in
any migration** yet is read/written by `platform/lambda/policies/main.py` and
altered by migration `007`. Treat its DDL as missing-from-source (see Gotchas).

## Components (the SQL files, in apply order)

### `platform/sql/schema.sql` — Phase 0
- **Purpose:** tenancy + identity + audit baseline.
- **Creates:** `tenants`, `users`, `audit_events` + 2 indexes.

### `platform/sql/002_phase_a.sql` — Phase A
- **Purpose:** the core cloud-posture model.
- **Creates:** `cloud_connections`, `scans`, `findings`, `assets`, `events`,
  `drift_events`, `scores`, `feedback`, `llm_cache` + their indexes.

### `platform/sql/003_phase_f_risks.sql`
- **Creates:** `risks` (risk register, lifted from Shasta).

### `platform/sql/004_phase_ai.sql`
- **Creates:** `ai_connections`, `ai_assets`, `ai_relationships`, `ai_scans`.
- **Alters:** `findings ADD COLUMN evidence_packet JSONB` (`004:100`).
- Wrapped in `BEGIN/COMMIT`.

### `platform/sql/005_unified_entities.sql` — SP1
- **Creates:** `entities`, `edges` (the unified entity/edge graph that
  replaces the per-domain `*_assets`/`*_relationships` tables for app
  reads/writes).
- **Alters:** `findings ADD COLUMN subject_entity_id UUID REFERENCES entities(id)` (`005:48`).

### `platform/sql/006_conversations.sql` — SP4
- **Creates:** `conversations`, `conversation_messages` (chat-first front door).

### `platform/sql/007_approval_idempotency.sql` — SP4 4d
- **Alters:** adds `source_approval_id UUID` to **`risks`** and **`policies`**
  + partial-unique idempotency indexes on each.
- **No transaction wrapper** (Data API executes each statement individually).
- ⚠️ References the `policies` table, whose `CREATE TABLE` is absent from `sql/`.

### `platform/sql/008_findings_natural_key.sql`
- **Destructive:** `DELETE FROM findings` (purges all rows — legacy rows had
  hardcoded `domain='ai'` / `status='fail'` and cross-scan dupes).
- **Creates** unique index `findings_natural_key_idx` enabling the writer to
  UPSERT (`ON CONFLICT`) instead of insert-per-scan.

### `platform/sql/009_scan_tier.sql`
- **Alters:** `scans ADD COLUMN tier TEXT NOT NULL DEFAULT 'quick'` CHECK
  `(quick|medium|deep)`.

### `platform/sql/010_scan_phase.sql`
- **Alters:** `scans ADD COLUMN phase TEXT NOT NULL DEFAULT 'done'` CHECK
  `(region_discovery|first_signal|crown_jewel|full|done)`.

### `platform/sql/011_phase_soc.sql` — SOC Slice 1
- **Alters `events`:** adds 7 AI-enrichment cols + `mitre_technique`,
  `incident_id`, `source_event_id` (+ partial-unique idempotency index).
- **Alters `drift_events`:** adds `target_resource_arn`.

### `platform/sql/012_users_device_token.sql`
- **Alters:** `users ADD COLUMN device_token TEXT` (APNs push).

### `platform/sql/013_phase_soc_ti.sql` — SOC Slice 1c
- **Creates:** `threat_indicators` (global IOC substrate; no `tenant_id`).
- **Alters:** `events ADD COLUMN source_ip TEXT`.

### `platform/sql/014_ai_scans_repo_asset_fkey_to_entities.sql`
- **Alters:** repoints `ai_scans.repo_asset_id` FK from retired `ai_assets`
  to `entities(id)` with `NOT VALID` (leaves pre-migration rows as orphans).

### `platform/sql/015_mcp_connectors.sql` — MCP Connectors Slice 1
- **Creates:** `user_connectors`, `tenant_bot_connectors` (KMS-envelope OAuth
  tokens). Ensures `pgcrypto` extension.

## Gotchas / non-obvious behavior (every one verified against the SQL)

1. **`findings` PK is `finding_id`, not `id`** (`002:51`). The unified `entities`/`edges`/`ai_*` tables use `id`; `findings` predates them.
2. **`findings.status` has NO `open` value** (`002:59`) — it's `fail`/`pass`/`partial`/`not_assessed`/`not_applicable`. `risks.status` is the one that has `open` (`003:14`). Easy to conflate.
3. **`findings.frameworks` must be a JSON object `{}`, never an array** (`002:64`) — the iOS Finding decoder rejects arrays. Default is `'{}'`.
4. **`findings` has NO `kind` column.** `check_id` (`002:55`) is the rule identifier. Matcher SQL has paid for this twice.
5. **`findings.evidence_packet` (JSONB) is where extra JSON goes — NOT `attributes`** (`004:100`). `findings` has no `attributes` column at all; `entities`/`edges`/`ai_assets` do.
6. **`findings.conn_id` is NOT NULL and FKs to `cloud_connections.conn_id`** (`002:53`), not `ai_connections`. AI-side synthetic findings must borrow a real cloud connection. `findings.scan_id` FKs to `scans.scan_id`, not `ai_scans`.
7. **Finding dedup is an UPSERT on `findings_natural_key_idx`** = `(tenant_id, conn_id, check_id, COALESCE(resource_arn,''), COALESCE(region,''))` (`008:20`). Don't INSERT without `ON CONFLICT` or you'll get duplicate-per-scan rows (the exact bug 008 fixed, after a `DELETE FROM findings`).
8. **`entities` PK is `id`, not `entity_id`** (`005:7`). **No `parent_id` column** — containment is an `edges` row (`edges.kind` = `uses`/`imports`/`runs`/…).
9. **`edges` uses `source_entity_id` / `target_entity_id`** (`005:30-31`), NOT `source_id`/`target_id`.
10. **`threat_indicators.indicator_value`, not `value`** (`013:8`), and it has a **composite PK** `(indicator_value, kind, source)` and **no `tenant_id`** (global table by design).
11. **`ai_connections` / `ai_scans` PK is `id`** and `ai_scans.status` uses **`success`** (not `completed`) (`004:80,85`). `ai_connections.status` uses `failed` (not `error` like cloud connections).
12. **`ai_scans.repo_asset_id` now FKs to `entities.id`** with `NOT VALID` (`014:23`), repointed from the retired `ai_assets`. Pre-migration rows are orphans referencing nothing.
13. **`policies` has NO `CREATE TABLE` anywhere in `platform/sql/`.** It is read/written by `platform/lambda/policies/main.py` and altered by `007`. Its DDL was applied out-of-band (manual `execute-statement` or an unversioned script). A spec author grepping `sql/` will wrongly conclude it doesn't exist. Inferred columns: `policy_id, tenant_id, template_key, title, content_md, soc2_controls, vars, status, version, created_at, updated_at, source_approval_id`. **Recommendation: backfill a migration capturing its real DDL.**
14. **Most enums are comment-only, not DB CHECK constraints** (findings, scans.status, events, tenants, cloud_connections, feedback, threat_indicators). The DB will accept any TEXT; values are enforced only by app code. Only `risks`, `ai_*`, `entities.domain`, `conversation_messages.role`, `scans.tier/phase` have real CHECKs.
15. **The Aurora Data API runs one statement per call** and doesn't support multi-statement transactions — migrations `007`, `008` etc. note this; some wrap `BEGIN/COMMIT` (single-call sends are fine) and some deliberately don't (`007`). Don't assume atomicity across `ALTER`s in a migration file.
16. **`drift_events` is a 1:1 extension of `events`** keyed on `event_id` (`002:120`), read via `LEFT JOIN drift_events USING (event_id)`. It is NOT a standalone event log.
17. **`risks.finding_id` has no FK constraint** (`003:17`) — it's a soft link; deleting a finding won't cascade.

## Optimization opportunities

- **Backfill the `policies` DDL into a versioned migration** (Gotcha 13). A
  fresh-cluster restore from `sql/` alone would be missing the table and the
  policies Lambda would 500.
- **Drop the 6 dead tables** (`llm_cache`, `audit_events`, `assets`, `scores`,
  `ai_assets`, `ai_relationships`) once confirmed with prod — they carry
  indexes and FKs that constrain nothing useful. `ai_assets`/`ai_relationships`
  still hold orphan FKs referenced by retired code paths; dropping needs the
  `014` orphan-row handling considered.
- **Promote comment-only enums to CHECK constraints** for the high-traffic
  tables (`findings.status`, `findings.severity`, `scans.status`) — the
  iOS/app currently bears all enforcement, and the `008` migration exists
  precisely because bad hardcoded values (`domain='ai'`, `status='fail'`)
  leaked into `findings`.
