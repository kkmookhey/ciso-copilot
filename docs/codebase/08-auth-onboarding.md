> Generated by codebase-documentation sweep 2026-06-05. Purpose: give a

## Overview

This subsystem owns three distinct flows, all backed by Aurora (via the RDS
Data API) and fronted by API Gateway + a Cognito user pool:

1. **Auth / tenant lifecycle** — Cognito post-confirmation creates a
   **per-user-or-per-domain tenant** in `pending` status, then an admin
   approves/rejects it (via signed email link *or* in-app), then iOS
   registers an APNs device token.
2. **Multi-tenant Microsoft IdP routing** (`auth_discover`) — an unauthed
   pre-login endpoint that lazily provisions a per-Entra-tenant Cognito
   OIDC IdP and returns the `authorize_url` so the client skips the Cognito
   Hosted-UI picker.
3. **Cloud onboarding** — four `initiate`/`complete` (or `callback`) pairs
   for AWS, Azure, GCP, Entra. `initiate` is JWT-authed and creates a
   `pending` `cloud_connections` row + a one-time `external_id`/`state`;
   `complete`/`callback` is **not** JWT-authed (authenticated by the
   one-time secret echoed back) and flips the row to `active`.

All handlers are Python 3.12 zip Lambdas (`main.handler`), wired in
`platform/lib/api-stack.ts` and `platform/lib/auth-stack.ts`. Schema lives in
`platform/sql/schema.sql` (tenants, users), `002_phase_a.sql`
(cloud_connections, scans), `012_users_device_token.sql` (device_token).

## The three flows

### (a) Auth / tenant lifecycle
1. User federates (Microsoft/Google) → Cognito fires **post_confirmation**.
2. `_resolve_identity` extracts `(sso_provider, sso_subject)`. `_tenant_key`
   decides the tenant identity: **per-user** (full email) for personal-email
   domains, **per-domain** for corporate domains.
3. If no tenant for that key → INSERT `tenants` (`status='pending'`), UPSERT
   `users` (`role='admin'`), sign an HS256 approve/reject JWT (7-day exp,
   single-use nonce), SES-mail it to `APPROVAL_RECIPIENT`. Else → UPSERT the
   user as `role='member'` on the existing tenant.
4. **Approval** — two paths to the same `tenants.status` flip:
   - Email link → `admin_decision` (`GET …/decision?token=`, JWT-verified,
     nonce appended to `approval_token_nonces`).
   - In-app → `admin_tenants` (`POST …/action`, Cognito-authed, `ADMIN_EMAILS`
     gated).
   Gating is enforced *elsewhere* (the API authorizer / handlers reject when
   `tenant.status != 'approved'`), so a pending user can sign in but can't use
   most endpoints.
5. iOS → `device_token_register` stores the APNs token on `users.device_token`.

### (b) Multi-tenant Microsoft IdP routing
`auth_discover` runs **before** OAuth. Google domains short-circuit to the
built-in `Google` IdP. Microsoft domains resolve the Entra tenant GUID via
OIDC discovery, then `_create_microsoft_idp` lazily creates a Cognito OIDC IdP
named `MS-<29hex>` pointed at `https://login.microsoftonline.com/<tenant>/v2.0`
and attaches it to both app clients. Returns an `authorize_url` with
`identity_provider=<name>` so the client bypasses the Hosted-UI picker.
Onboarding a new customer Entra tenant = their admin consents to our app reg +
their first user hits this endpoint.

### (c) Cloud onboarding — what initiate + complete do
| Cloud | initiate returns | complete/callback auth | credential model | `account_identifier` |
|-------|------------------|------------------------|------------------|----------------------|
| **AWS** | CFN Console deep-link + presigned S3 template URL | `external_id` in body | STS assume-role + external_id (secret) | AWS account id |
| **Azure** | Cloud Shell `curl\|bash` `run_command` | `external_id` in body | SP `client_id`+`client_secret` (secret) | Entra tenant id |
| **GCP** | Cloud Shell `curl\|bash` `run_command` | `external_id` in body | Workload Identity Federation (sa_email/wif_pool/wif_provider in `scope`, no secret) | project_id or org_id |
| **Entra** | Microsoft admin-consent URL | `state` in query (stored as `external_id`) | our app client-credentials + customer tenant id (no tokens stored) | Entra tenant id |

All `initiate` handlers: JWT-authed, INSERT pending `cloud_connections` row,
generate `secrets.token_urlsafe(24)` one-time secret. All `complete`/`callback`
handlers: unauthed at the gateway, verify the one-time secret + `status=='pending'`
(409 if already completed), UPDATE row to `active`. **None auto-trigger a scan**
(Slice 2b) — the user starts scans from `/scan`.

## sso_subject / identities[0].userId pattern (as implemented)

The gotcha is correctly implemented in **every** handler that needs the
upstream IdP subject. `users.sso_subject` stores `identities[0].userId` (the
upstream IdP sub), **not** `claims.sub` (the Cognito-pool sub):

- **Set on write:** `post_confirmation/main.py:132-157` `_resolve_identity` —
  parses the `identities` attribute (JSON array string), reads
  `ids[0]['userId']` as the subject and `ids[0]['providerName']` as the
  provider; per-tenant `MS-*` provider names normalize to `"microsoft"`. Falls
  back to `("cognito", attrs['sub'])` only when there is no `identities` attr.
  `_upsert_user` writes it to `users.sso_subject` (line 213).
- **Read on lookup:** identical `identities[0].userId or claims.sub` logic in
  `device_token_register/main.py:61` (`_resolve_subject`),
  `onboarding_aws_initiate/main.py:138` (`_subject_from_claims`), and inline in
  `onboarding_azure_initiate:86`, `onboarding_gcp_initiate:76`,
  `onboarding_entra_initiate:91`. Each then `SELECT tenant_id FROM users WHERE
  sso_subject = :s`.

Reaching for `claims.get("sub")` directly would silently 401 every federated
user (matches the CLAUDE.md canonical rule).

## Optimization opportunities

- **Subject-resolution duplication** — the `identities[0].userId or sub`
  block + `SELECT tenant_id FROM users WHERE sso_subject` is copy-pasted across
  5 handlers (`device_token_register`, all four `*_initiate`, plus
  `post_confirmation`). A shared helper module would remove ~30 lines ×5 and
  guarantee the gotcha stays consistent. Concrete, low-risk.
- `admin_tenants._list_tenants` (`main.py:55-62`) and `admin_decision._get_admin_user_email`
  run a correlated subquery `(SELECT u.email FROM users … ORDER BY created_at
  LIMIT 1)` per tenant row — fine at current scale (single-digit pending
  tenants) but an N+1 if the pending list ever grows; a `DISTINCT ON
  (tenant_id)` join would flatten it.
- `auth_discover._ensure_idp_on_single_client` (`main.py:165`) does a
  `describe_user_pool_client` + conditional `update_user_pool_client` on
  **every** discover call even when the IdP is already attached. The describe
  is cheap but the call runs unauthenticated and pre-login; a short-lived
  in-memory cache of "client already has IdP X" would cut two Cognito API
  calls per repeat login.

## Gotchas / non-obvious behavior

- **Per-user tenant model (load-bearing).** `post_confirmation._tenant_key`
  (line 77) keys personal-email domains on the *full email*, corporate domains
  on the domain. This is the fix for the multi-tenant leak where every
  `gmail.com` user auto-joined the first gmail tenant as a `member`. Matches
  MEMORY: domain-sharing was removed 2026-06-03. **`tenants.email_domain` is
  UNIQUE and semantically overloaded** — it is the tenant identity key, not
  necessarily a domain. A spec author must NOT assume colleagues at `acme.com`
  *do* share a workspace just because corporate domains key on domain — there
  is no invite/team flow; corporate-domain sharing is incidental, not a
  product feature.
- **`external_id` column doubles as the Entra `state`.** Entra has no
  external_id concept; the one-time consent `state` is stored in
  `cloud_connections.external_id` and looked up there
  (`onboarding_entra_callback:88`).
- **Two parallel approval paths, one source of truth.** Email-link
  (`admin_decision`, HS256-token, single-use nonce in `approval_token_nonces`)
  and in-app (`admin_tenants`, Cognito + `ADMIN_EMAILS`). Both flip
  `tenants.status`. Don't add a third without reconciling the nonce logic.
- **`admin_tenants` route is `/action`, not `/decision`.** The handler
  docstring is stale (`api-stack.ts:556-559` mounts `/admin/tenants/{id}/action`).
- **`complete`/`callback` endpoints are unauthenticated at the gateway** and
  rely entirely on the one-time `external_id`/`state` + `status=='pending'`
  guard. AWS also verifies `external_id` matches before activation
  (`onboarding_aws_complete:61`).
- **Credentials are heterogeneous per cloud:** AWS = STS role+external_id
  (secret), Azure = SP client secret (secret), GCP = WIF descriptor in
  `scope` (NO secret stored — `credentials_secret_arn` keeps its placeholder),
  Entra = none (our app client-credentials at scan time). A spec assuming "all
  connections have a real secret" is wrong for GCP and Entra.
- **No auto-scan on onboarding** (Slice 2b). Every `complete`/`callback`
  returns `initial_scan_id: None`; a fresh connection shows "Never scanned"
  until the user triggers `/scan`.
- **`MS_PERSONAL_TENANT`** (`post_confirmation:51`,
  `9188040d-6c67-4c5b-b112-36a304b66dad`) is a backstop reject for personal
  Microsoft accounts; the `/organizations` issuer is the primary guard.
