## Overview

The entire AWS footprint of Shasta/CISOBrief is one CDK app (TypeScript,
`aws-cdk-lib` ^2.165) in `platform/`, entry `platform/bin/platform.ts`.
It synthesizes **8 stacks** (network → data → auth → ecr → static → events
→ scan → api), all deployed to one account/region (`us-east-1` default, from
`config.awsRegion`). The API stack is an API Gateway REST API (stage `v1`)
fronting **~30 Python/container Lambdas**, Cognito-authed via a User Pool
authorizer. Real-time customer events fan into a single EventBridge bus →
router Lambda. Scans run as ECS Fargate tasks (AWS/Azure/GCP) plus container
Lambdas (Entra, AI). Customer onboarding is driven by artifacts in
`platform/cfn/` (a CloudFormation template for AWS, bash scripts for Azure +
GCP) served from a CloudFront CDN.

There are 9 files in `platform/lib/` but only **8 stacks** — `config.ts` is
shared config, not a stack. (The prompt's "9 stacks incl. config" is the
file count, not the stack count.)

## Components

### `platform/bin/platform.ts`
- **Purpose:** CDK app entry — instantiates and wires the 8 stacks.
- **Entry point:** top-level `new cdk.App()` (line 14).
- **Stack order + cross-stack props (lines 22–79):**
  - `CisoCopilotNetwork` → exports `vpc`.
  - `CisoCopilotData` (needs `vpc`) → `cluster`, `connectorTokensKey`,
    `pkceVerifierTable`, `autonomousBroadcastQueue`, `autonomousBroadcastSeenTable`.
  - `CisoCopilotAuth` (needs `dbCluster`) → `userPool`, `userPoolClient` (iOS),
    `webClient`.
  - `CisoCopilotEcr` → 6 ECR repos.
  - `CisoCopilotStatic` → `cdnDistribution` (+ unused-by-CDK app dist).
  - `CisoCopilotEvents` (needs `dbCluster`) → `eventBus`.
  - `CisoCopilotScan` (needs vpc, dbCluster, 5 ECR repos, broadcast queue/table,
    connectorTokensKey) → scanner Lambdas + Fargate task defs + secrets +
    `aiScanQueue` + `scanCluster`.
  - `CisoCopilotApi` (last — needs nearly everything).
- **Gotcha (lines 56–67):** task-def **family names are passed as plain
  strings** (`'ciso-copilot-aws-scan'`, `-azure-scan`, `-gcp-scan`), NOT ARNs,
  to dodge a cross-stack CFN export on a revision ARN that changes every
  task-def update. Role ARNs (stable) are passed as live props.

- **Purpose:** the only VPC. 2 AZs, **1 NAT gateway**, three subnet tiers:

  all /24 (lines 11–19).

### `platform/lib/data-stack.ts` (`CisoCopilotData`)
- **Purpose:** all persistent data + the autonomous-broadcast pipeline plumbing.
- **Resources:**
  - **KMS `StorageKey`** (line 27) — Aurora encryption, rotation on.
  - **Aurora PostgreSQL Serverless v2** `AuroraPg` (line 32): engine **VER_16_4**,
    isolated subnets, writer `serverlessV2` min 0.5 / max 4 ACU, DB name
    `ciso_copilot`, **Data API enabled** (line 43), 30-day backup,
    removalPolicy SNAPSHOT.
  - **KMS `ConnectorTokensKey`** (line 48), alias `cisocopilot-connector-tokens`
    — envelope key for MCP connector OAuth tokens (per-row pgcrypto). RETAIN.
  - **DynamoDB `PkceVerifierTable`** (line 55) `cisocopilot-pkce-verifiers`,
    PK `nonce`, TTL `ttl`, PAY_PER_REQUEST, removalPolicy DESTROY.
  - **SQS `AutonomousBroadcastDlq`** (line 65, 14-day retention) +
    **`AutonomousBroadcastQueue`** (line 71, 30s visibility, 4-day retention,
    maxReceiveCount 5).
  - **DynamoDB `AutonomousBroadcastSeen`** (line 82) — per-finding dedup, PK
    `seen_key`, TTL `ttl_epoch`, PAY_PER_REQUEST.
  - **2 CloudWatch alarms:** `AutonomousBroadcastDlqAlarm` (line 90, DLQ has a
    message 5×1min) and `AutonomousBroadcastDriftAlarm` (line 104, math
    expression `CriticalFailWritten - BroadcastQueued > 2/h` over namespace
    `Shasta/AutonomousBroadcast`).
- **Exports:** `cluster`, `storageKey`, `connectorTokensKey`,
  `pkceVerifierTable`, `autonomousBroadcastQueue`, `autonomousBroadcastDlq`,
  `autonomousBroadcastSeenTable`.

### `platform/lib/api-stack.ts` (`CisoCopilotApi`)
- **Purpose:** the REST API + ~22 Lambdas behind it + the streaming chat URL +
  tools dispatcher + connectors OAuth. See full route table below.
- **API:** `RestApi` `ciso-copilot` (line 454), stage `v1`, INFO logging,
  default CORS preflight ALL_ORIGINS/ALL_METHODS. Gateway responses 4xx/5xx/401/

  real errors instead of CORS bounces.
- **Authorizer:** `CognitoUserPoolsAuthorizer` over `props.userPool` (line 481);
  `authedOpts` applies it. UNAUTHED routes omit it (marked below).
- **Resource-limit note (line 67):** CDK soft cap lifted to 1000 via
  `@aws-cdk/core:stackResourceLimit` in `cdk.json`. Slice-2 admin connector
  routes still live OUT-OF-CDK (see `scripts/apigw-admin-routes-bootstrap.sh`).
- **Streaming:** `ChatStreamFn` (line 1021) runs the **Lambda Web Adapter**
  (LWA layer `…753240598075:layer:LambdaAdapterLayerX86:27`), handler `run.sh`,
  served via a **Lambda Function URL** (`RESPONSE_STREAM`, authType NONE,
  line 1039) — NOT an API Gateway route. URL exported as `ChatStreamUrl`.

### Env-var conventions
Every DB-touching Lambda gets `dbEnv` = `{DB_CLUSTER_ARN, DB_SECRET_ARN,
DB_NAME=ciso_copilot}`. APNs Lambdas get `APNS_PLATFORM_APP_ARN`. Anthropic
callers get `ANTHROPIC_SECRET_NAME=ciso-copilot/anthropic-api-key`. Connector/
voice/tools Lambdas get `CONNECTOR_TOKENS_KEY_ARN`.

### Customer onboarding artifacts (`platform/cfn/`)
- **`aws-onboard.yaml`** — the CFN template a customer deep-links into the AWS
  Console. Creates: `CISOCopilotReader` role (ReadOnlyAccess + SecurityAudit +
  ViewOnlyAccess, trust scoped by `ExternalId`, line 77); `CISOCopilotEventForwarder`
  role + `CISOCopilotForwardSecurityEvents` EventBridge rule on the **default**
  bus (line 122) forwarding GuardDuty/Inspector2/SecurityHub/Config/CloudTrail

- **`platform/lib/scan-stack.ts:85-95, 322-328`, `events-stack.ts:87-95`,

## Gotchas / non-obvious behavior

- **8 stacks, not 9.** `config.ts` is shared config. Stack names:
  `CisoCopilotNetwork/Data/Auth/Ecr/Static/Events/Scan/Api`.
- **Task-def family names are passed cross-stack as plain strings, never ARNs**
  (`bin/platform.ts:63-67`, used in `ecs:RunTask` + IAM `:*` wildcards). Don't
  "fix" this into an ARN export — it deliberately avoids a CFN export that
  changes on every task-def revision and can't be updated while imported.
- **The GCP scanner identity name `ciso-copilot-gcp-scanner` is load-bearing.**
  `scan-stack.ts:238` hard-codes it because the customer's WIF provider
  (`cfn/gcp/onboard.sh:66,166`) trusts that exact assumed-role name. Renaming the
  role breaks every onboarded GCP customer. Same role is BOTH the (retired) GCP
  Lambda role and the GCP Fargate taskRole (CompositePrincipal).
- **AWS posture scans run on Fargate, not Lambda.** The `ShastaRunner` Lambda
  exists but is not the prod path (see Dead code). New work that "adds an AWS
  scanner" already exists as `ciso-copilot-aws-scan` Fargate task.
- **Slice-2 admin connector routes are NOT in CDK.** They live in
  `scripts/apigw-admin-routes-bootstrap.sh` (5 routes under
  `/connectors/admin/slack/*`) because the ApiStack was at the 500-resource CFN
  limit when they were added. The CDK soft cap is now 1000 (`cdk.json`
  `stackResourceLimit`), but the admin routes were never migrated back in.
  Before adding routes, check this script too.
- **`/onboarding/*/complete`, `/onboarding/entra/callback`,
  `/connectors/callback/{kind}`, `/public/trust/{slug}`,
  `/admin/tenants/{id}/decision`, `/auth/discover-tenant` are UNAUTHED.** Each
  has its own gate (external_id in body, state-JWT in query, query-string token,
  or public-by-design). Don't assume Cognito covers every route.
- **Streaming chat is a Lambda Function URL, not an API Gateway route.** A spec
  that greps the route table for "streaming" finds nothing — it's
  `ChatStreamFn` + `addFunctionUrl(RESPONSE_STREAM)` via the LWA layer
  (`api-stack.ts:992-1047`). Managed Python runtimes can't stream through API
  Gateway; this is the existing workaround.
- **`ChatSessionFn` and `ChatStreamFn` share ONE source dir** (`lambda/chat_session`)
  with different handlers (`main.handler` vs LWA `run.sh`). `ChatSessionFn`
  explicitly excludes the streaming files (`app.py`, `run.sh`,
  `messages_stream.py`, `tools_dispatch.py`, `requirements.txt`) so its asset
  hash stays stable when streaming code changes (`api-stack.ts:936-943`).
- **The autonomous-broadcast pipeline already exists end-to-end** (Slice 2.4):
  scanners → `AutonomousBroadcastQueue` → `FindingsSubscriberFn` → Slack, with
  a dedup DDB table, SSM kill switch `/cisocopilot/autonomous_rule/enabled`, and
  two CloudWatch drift/DLQ alarms. Don't re-spec a "critical finding → Slack"
  feature.
- **SOC enrichment, threat-intel feeds (abuse.ch/KEV/Tor), per-tenant LLM spend
  cap, and APNs push all already exist** in the events stack. The spend table
  `soc_llm_spend_daily` doubles as the push rate-limit counter (different
  sort-key prefix, per its comment).
- **`feature-dev-ai-resources.yaml` is a test fixture, not product.** It
  deliberately ships insecure Bedrock/SageMaker/Lambda/S3 to make Shasta's
  AWS-AI checks fire. Don't read it as a reference architecture.
- **The CDN bucket auto-deploys `platform/cfn/` on every `CisoCopilotStatic`
  deploy** (`static-stack.ts:62`). Edits to the onboarding YAML/scripts ship by
  redeploying that stack — no separate upload step.
