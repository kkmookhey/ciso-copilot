# Connectors, Voice, Chat & Agentic Tools — Codebase Documentation

> Generated by codebase-documentation sweep 2026-06-05. Purpose: give a
> future spec author a grep-able answer to "does this MCP connector / voice
> / chat / agentic-tool capability already exist?" before they propose
> building it. Cite `file:line` everywhere.

## Overview

This subsystem covers everything between Shasta and outbound SaaS/LLM
integrations:

- **MCP connectors** (`platform/lambda/connectors/`, `_shared/mcp_oauth/`):
  per-user and per-tenant OAuth + remote-MCP token storage. **Slack is the
  only provider actually wired** (user-token + admin-bot flows). Atlassian /
  Google / Microsoft are stubbed in enums + comments only.
- **Voice** (`platform/lambda/voice_session/`, `chat_session/voice.py`):
  two distinct OpenAI Realtime ephemeral-key brokers. The backend is never
  on the audio path — it only mints a `client_secret` (`ek_...`).
- **Text chat / agentic loop** (`platform/lambda/chat_session/`):
  server-side Anthropic tool-use loop over a 12-tool catalog, streamed as
  SSE via Lambda Web Adapter.
- **Action tools** (`platform/lambda/tools/`): HTTP-dispatched
  side-effecting tools (Slack DM, JIRA, GitHub PR, Entra revoke, log
  search, forensic scan) used by the voice/iOS tool-call path.
- **`forensic_callback/`**: one-shot EventBridge target that fires the
  delayed "scan done" APNs push.
- **`_shared/`**: reusable helpers vendored into many Lambdas
  (broadcast fan-out, MCP OAuth package, TI lookup, IOC extraction,
  GreyNoise, push, speakable labels, MCP stdio client).

Triggers: API Gateway REST (`/v1/...`, `/connectors/...`), a Lambda
Function URL (`/v1/conversations/{id}/stream`, RESPONSE_STREAM), an SQS
subscriber (autonomous broadcast, see findings_subscriber — out of scope),
and a one-time EventBridge rule (forensic callback).

## MCP connector model

### Data model — migration `015_mcp_connectors.sql`

Two tables (`platform/sql/015_mcp_connectors.sql`):

- **`user_connectors`** (`015_mcp_connectors.sql:11`) — one active row per
  `(tenant_id, user_id, oauth_provider)` (UNIQUE at `:30`). Columns:
  `conn_id` PK, `mcp_server_url`, `vendor_user_id`, `vendor_workspace_id`,
  `access_token_enc` + `access_data_key_ct`, `refresh_token_enc` +
  `refresh_data_key_ct`, `access_expires_at`, `scopes TEXT[]`, `status`
  (`active`/`revoked`/`error`/`expired`), `last_error`, `last_used_at`,
  `revoked_at`. Partial indexes on the active-lookup and refresh-by-expiry
  paths (`:33`, `:36`).
- **`tenant_bot_connectors`** (`015_mcp_connectors.sql:41`) — admin-installed
  workspace bot, one per `(tenant_id, oauth_provider)` (UNIQUE at `:61`).
  Adds `broadcast_channel_id`, `broadcast_channel_name`,
  `autonomous_rule_enabled` (default `true`), `installed_by_user_id`.
  Refresh columns are nullable here (Slack bot tokens don't rotate).

**KMS-envelope encryption** (`_shared/mcp_oauth/crypto.py`): every token
column is paired with a `*_data_key_ct` column. `encrypt_token()` (`:69`)
calls `kms.GenerateDataKey` per call → Fernet-encrypts with the plaintext
key, persists `(fernet_ct, kms_ciphertext_blob)`. `decrypt_token()` (`:90`)
`kms.Decrypt`s the blob (memoized in a bounded 256-entry LRU keyed by
SHA256(blob), `:42`–`:66`). **Both columns are required to decrypt** — the
module docstring (`:18`) records the prior broken design (one data key per
cold start) that bricked connectors under concurrency.

### OAuth flow primitives (`_shared/mcp_oauth/`)

- **`pkce.py`** — RFC 7636 PKCE. `generate_pair()` (`:11`), verifier stored
  in DynamoDB (`cisocopilot-pkce-verifiers`, env `PKCE_VERIFIER_TABLE`)
  with 300s TTL; `fetch_verifier()` (`:43`) consumes one-shot via
  `delete_item ReturnValues=ALL_OLD` (atomic; concurrent callbacks see it
  at most once).
- **`state.py`** — signed-JWT state parameter **is the CSRF defense** (the
  earlier double-submit cookie was dead code cross-origin; docstring `:1`).
  HS256, `iss=shasta-connectors`, `aud={provider}-callback` (`:39`),
  5-minute window. `verify_state(token, expected_provider=...)` (`:68`)
  pins the audience so a Slack JWT can't replay at another callback.
  Secret `STATE_JWT_SECRET` must be ≥32 bytes (`:43`).
- **`providers/slack.py`** — the only provider config. User scopes
  `chat:write,im:write,im:history,search:read,users:read` (`:7`);
  `MCP_SERVER_URL=https://mcp.slack.com/mcp` (`:9`). `build_authorize_url`
  (`:14`, parametrized scope/user_scope so the same builder serves user +
  bot flows), `exchange_code` (`:39`, prefers `authed_user.*` for user
  tokens), `exchange_code_bot` (`:82`, top-level `xoxb-` bot token),
  `refresh_token` (`:114`).
- **`session.py`** — runtime resolution + JIT refresh + MCP session for
  **per-user** connectors. `_DataAPIWrapper` (`:43`) wraps Aurora Data API
  with an explicit `transaction()` context manager (`:69`).
  `refresh_if_near_expiry()` (`:214`) is the **tx-scoped advisory-lock**
  pattern: NULL-safe expiry check, then `pg_advisory_xact_lock(hash(conn_id))`
  inside one Data API transaction (`:241`) so the lock spans the re-read +
  provider HTTP refresh + UPDATE. Without the explicit transaction the Data
  API auto-commits and the lock releases instantly (docstring `:62`,
  `:226`). `get_session(subject, kind, *, tenant_id)` (`:308`) opens the
  remote MCP `streamablehttp_client` with a Bearer token. `discover_tools`
  (`:393`) fans out `list_tools()` across all of a user's active connectors
  (5-min cache keyed by workspace+scopes hash, `:343`).
- **`admin_session.py`** — mirror of `session.py` for
  `tenant_bot_connectors`. `lookup_tenant_bot()` (`:25`),
  `get_admin_session()` (`:49`, no refresh path — Slack bot tokens don't
  expire). Consumed by `findings_subscriber/main.py` (autonomous
  broadcast) and the admin channel picker.

| DELETE | `/admin/slack` | `…revoke_workspace_bot:168` | admin |
| GET | `/admin/slack/status` | `…admin_bot_status:193` | admin |

**`/admin/slack/*` routes were bootstrapped OUT of CDK** (commit
`096d81d`, 2026-06-02): `CisoCopilotApi` was at 498 resources after the
Slice 2.4 broadcast pipeline; the 7 admin routes would push past CFN's hard
500-resource-per-stack limit. They are provisioned by
`platform/scripts/apigw-admin-routes-bootstrap.sh` (idempotent, raw
`aws apigateway` CLI). **The Lambda's internal `_route` dispatcher already
handles every admin path** — only the API Gateway resources are out-of-CDK.
A future CDK redeploy that recreates `/connectors` requires re-running that
script (manual steps documented in the commit body).

**Tenant/user resolution** is the canonical `sso_subject` JOIN, NOT a JWT
`custom:tenant_id` claim (which this Cognito pool does not issue).
`subject_from_claims()` (`main.py:23`) prefers `identities[0].userId` over
`claims.sub` for federated users. `_resolve_user_context()`
(`handlers_slack.py:14`) and `_require_admin()`
(`handlers_slack_workspace_bot.py:25`, adds `AND role='admin'`) are the JOIN
helpers. Revoke (`handlers_common.py:16`) scopes by
`(tenant_id, user_id, conn_id)` and 404s on miss to avoid intra-tenant IDOR.

**Admin channel picker uses Slack's direct Web API, not MCP**
(`handlers_admin_slack_channels.py:13` docstring): Slack's MCP server
rejects bot (`xoxb-`) tokens with 401 — it only accepts user (`xoxp-`)
tokens. So `conversations.list` / `auth.revoke` go through
`https://slack.com/api/*` directly with the decrypted bot token
(`_bot_token:32`, `_slack_get:49`, `_slack_post:64`).

**Secrets** (`_secrets.py`): `SLACK_CLIENT_ID`, `SLACK_CLIENT_SECRET`,
`STATE_JWT_SECRET` are SSM SecureStrings loaded into `os.environ` at
cold-start import (CFN can't inject SecureString as Lambda env vars). The
tools/voice Lambdas lazy-load the same Slack creds via
`session._ensure_provider_secrets()` (`:172`).

### Which connectors are wired vs stubbed

- **Slack — SHIPPED** end-to-end: per-user OAuth, admin workspace-bot
  install, channel picker, broadcast-channel config, autonomous-rule toggle,
  revoke. (Matches MEMORY: PRs #34/#35/#36/#38/#39/#40 + hotfixes.)
- **Atlassian / Google / Microsoft — STUBBED**. They appear only in the
  `ProviderKind` Literal (`session.py:19`), `_MCP_PROVIDER_KINDS`

  No provider module, no `connect/{kind}` route, no `_provider_refresh`
  branch (`session.py:202` raises `NotImplementedError` for non-Slack).

## Agentic tool inventory

There are **two separate tool registries** — the text-chat catalog (read +
propose tools, in-process, against Aurora) and the action-tool dispatcher
(side-effecting, HTTP, used by voice/iOS). Plus a third (the voice session's
native tool *definitions*). They overlap by intent but are distinct code.

### A. Text chat catalog — `chat_session/tools_dispatch.py` (12 tools)

Server-side Anthropic tool-use loop (`app.py:127`, max 6 rounds). Every
tool is tenant-scoped against Aurora via `_db._q`; the 8 data tools
replicate the SQL of the REST Lambdas rather than HTTP-calling them
(`tools_dispatch.py:17`). `dispatch(name, tenant_id, args)` (`:824`),
schemas via `anthropic_tool_defs()` (`:812`).

| `get_entity` (`:410`) | data | Single `entities` row by UUID. |

## Voice session — key facts

Two brokers, both POST to `https://api.openai.com/v1/realtime/client_secrets`
(GA shape; the Beta `/v1/realtime/sessions` endpoint was retired). Backend
is never on the audio path — it returns the ephemeral key (`value`,
prefix `ek_`) and the iOS/web client opens the media channel directly.

- **`voice_session/main.py`** (`POST /voice/session`, the iOS path):
  - **Model `gpt-realtime` pinned** (`:62`). **No `session.temperature`** —
    OpenAI Realtime GA (2026) removed it; tone = voice + system prompt
    (`:94`).
  - **`turn_detection.interrupt_response: True`** (`:84`) — required, else
    every user barge-in 400s with "active response in progress".
  - `server_vad`, voice `coral`, PCM 24 kHz, whisper-1 transcription.
  - System prompt from `system_prompt.py` (persona-only Shasta prompt,
    rendered with first name + connected clouds); native + per-user MCP
    tools merged via `_build_openai_tools_sync` (`:372`).
  - `_subject_from_claims` (`:418`) is the **canonical** federated-subject
    impl (`identities[0].userId` then `sub`) — copied verbatim into
    connectors, tools, and chat_session.
- **`chat_session/voice.py`** (`POST /v1/conversations/{id}/voice`, the web
  path): **different model `gpt-realtime-2`** (`:33`), voice `alloy`,
  **`interrupt_response: False`** (`:113` — web barge-in handled
  browser-side via `response.cancel`), VAD threshold 0.6 / 700 ms silence
  (whisper-hallucination mitigation). Tools come from the POST body
  (browser supplies `tools.ts` definitions). Response key is **`value`**,
  not `client_secret` (`:151`) — the web voiceClient reads `.value`.

> Surprising: the iOS broker pins `gpt-realtime` with `interrupt_response:
> True`; the web broker pins `gpt-realtime-2` with `interrupt_response:
> False`. The CLAUDE.md "one model version pinned in one config value" rule
> is **not** currently honored across these two files.

## Optimization opportunities

1. **`refresh_if_near_expiry` holds the provider HTTP call inside the Aurora
   transaction** (`session.py:275`–`:305`). This is *intentional* (the
   advisory lock must span the refresh decision), but it parks an open
   Postgres transaction for the ~2s Slack round-trip. With many concurrent
   users on the same connector this serializes refreshes. Acceptable now;
   revisit if connector volume grows (consider a short lock + optimistic
   compare-and-set on `access_expires_at` instead).

2. **`get_finding` can full-table-scan up to 200 findings in Python**
   (`tools_dispatch.py:473`): when a `finding_id` is supplied without a
   `check_id`, it pulls `limit=200` findings and filters in a Python `next()`
   loop. A direct `WHERE finding_id = :id` query (tenant-scoped) would be a
   single indexed lookup.

3. **`mcp_client._invoke_mcp_tool` spins a fresh asyncio loop + stdio
   subprocess per call** (`mcp_client.py:50`). For `create_jira_ticket` this
   means cold-starting the Atlassian MCP server (`npx`-style) on every
   invocation — high per-call latency. If JIRA usage grows, a warm session
   or the streamable-HTTP `mcp_oauth` path would amortize it. (Also note the
   `HOME=/tmp` / `EROFS` Lambda gotcha from CLAUDE.md applies to any
   `npx`-based stdio MCP server here.)

## Gotchas / non-obvious behavior

- **Cognito subject extraction**: every handler that JOINs `users.sso_subject`
  MUST use `subject_from_claims` (prefers `identities[0].userId`, falls back
  to `sub`). `claims.get("sub")` directly silently 401s every federated
  Microsoft/Google user. There is **no `custom:tenant_id` claim** in this
  pool — tenant is always a DB JOIN.
- **`/admin/slack/*` API Gateway resources live outside CDK** (commit
  `096d81d`, CFN 500-resource cap). The Lambda handles the paths; only the
  gateway plumbing is bootstrapped via
  `platform/scripts/apigw-admin-routes-bootstrap.sh`. Re-run after any CDK
  deploy that recreates `/connectors`.
- **Slack MCP rejects bot tokens** — admin/bot Slack operations use direct
  Slack Web API, only per-user (`xoxp-`) tokens go through `mcp.slack.com`.
- **KMS envelope: both `*_token_enc` and `*_data_key_ct` are required.**
  Losing the data-key column makes the token unrecoverable. Don't add a
  token column without its paired data-key column.
- **tx-scoped advisory lock only works inside an explicit Data API
  transaction** — `pg_advisory_xact_lock` releases on auto-commit. Use
  `_db().transaction()`, never bare `execute`, for any lock-protected
  read-modify-write.
- **Aurora Data API rejects array input parameters** — bind PG array
  literals as strings and cast `:scopes::text[]` (`handlers_slack.py:131`).
- **Two voice brokers diverge** on model + `interrupt_response` (iOS:
  `gpt-realtime`/True; web: `gpt-realtime-2`/False). Don't assume one.
- **Text agentic loop does not replay tool turns**: `_history_for_anthropic`
  (`app.py:73`) replays only user/assistant *text* — stored `tool` messages
  are UI artifacts, and tool_use/tool_result blocks are never persisted, so
  the model re-derives tool calls fresh each turn. Loop caps at
  `MAX_TOOL_ROUNDS = 6` (`app.py:48`).
- **`propose_*` tools never mutate** — they return an `approval_card`; the
  actual write happens later when the user approves (PATCH
  `/messages/{id}`). Don't add a "create risk directly" tool.
