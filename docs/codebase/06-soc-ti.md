## Overview

This subsystem is the real-time SOC pipeline. Every event landed on the
central EventBridge bus (GuardDuty / Inspector2 / SecurityHub findings +
CloudTrail management-API calls + Config item changes) fans into a single
**event_router** Lambda, which resolves the tenant, normalizes the event,
archives the raw payload to S3, inserts an `events` row (+ a `drift_events`
row for config changes), fires APNs push for high/critical severities, and
— for drift only — enqueues an SQS message for async AI enrichment.

**soc_enrichment** consumes that queue, computes statistical features
(off-hours, first-time-actor, action-rarity, blast-radius, threat-intel
matches), calls an LLM (LiteLLM → Claude Sonnet) under a per-tenant daily
spend cap, and writes `ai_*` columns back onto the `events` row.

**events_list** is the read API (`GET /events`, `GET /events/{id}`,
`POST /events/{id}/feedback`) that surfaces these rows to web/iOS.

Three cron **ti_feed_*** Lambdas (abuse.ch hourly, CISA KEV daily, Tor
hourly) populate the global `threat_indicators` table that enrichment reads.

**findings_subscriber** is a separate autonomous-broadcast path (not part of
the event pipeline): scanners publish critical-fail findings to an SQS queue;
this Lambda decrypts the tenant's Slack bot token and posts a Block Kit card.

Triggers: EventBridge catch-all rule (router), SQS (soc_enrichment +
findings_subscriber), EventBridge cron (ti_feed_*), API Gateway (events_list).

## The `detail-type` (not `source`) gotcha — CONFIRMED

CLAUDE.md warns: EventBridge "AWS API Call via CloudTrail" events arrive with
`source: aws.<service>` (aws.ec2/aws.iam/aws.s3), never `aws.cloudtrail`, so
the router must key on `detail-type`. Confirmed in **all four** named functions
and at the infra level:

- `_normalize()` (main.py:238, 246) — `if detail_type == "AWS API Call via CloudTrail"` / `"Configuration Item Change Notification"`. Comment at lines 235-237 documents exactly this. GuardDuty/Inspector2/SecurityHub branch on `source` because those DO arrive as `aws.guardduty` etc.
- `_classify_kind()` (main.py:293) — `if detail_type in ("AWS API Call via CloudTrail", "Configuration Item Change Notification")`.
- `_extract_states()` (main.py:300, 310) — branches on `detail_type` only.
- `_source_event_id()` (main.py:322, 324) — branches on `detail_type` first (`detail.eventID` / `captureTime:resourceId`), THEN source for the native-detector sources.
- **Infra level:** the EventBridge rule `fan-all-to-router` (events-stack.ts:240-243) is a **catch-all** matching `account: {exists: true}` — it does NOT filter on `source` at all, so no real events are dropped at the rule. The whole detail-type discrimination happens in code.

## findings_subscriber broadcast flow

Two halves, decoupled by SQS:

1. **Producer** (`_shared/broadcast_fanout.publish_if_critical`): scanner
   writers call this for each finding. It emits `CriticalFailWritten` (always,
   for drift detection), and if `severity=='critical' AND status=='fail'` AND
   `AUTONOMOUS_BROADCAST_QUEUE_URL` is set, publishes `{tenant_id, finding_id,
   scan_id}` to SQS (emits `BroadcastQueued`). Failures swallow + log +
   `BroadcastFanoutFailed`. CloudWatch alarm watches
   `(CriticalFailWritten - BroadcastQueued) > 2/hr`.

2. **Consumer** (`findings_subscriber/main.py`): SQS batch=1 →
   `idempotency.seen()` (DDB, 7d TTL) → `kill_switch.global_enabled()` (SSM,
   fail-open) → `tenant_bot_connectors` gate (silent ack if missing / disabled
   / no channel) → re-read `findings` (silent ack if missing) →
   `block_kit.format_finding_card` → KMS-decrypt the bot token
   (`mcp_oauth.crypto.decrypt_token` of `access_token_enc` + `access_data_key_ct`)
   → POST `https://slack.com/api/chat.postMessage` with `Bearer <bot_token>`
   and the Block Kit blocks → `mark_seen` (swallowed if it fails post-send, to
   avoid double-broadcast). Card deep-links `{WEB_BASE_URL}/risks/{finding_id}`.

**Recent fixes (verify before re-touching):**
- `e14b1f5` — broadcast moved **post-commit, not in-tx**. `publish_if_critical`
  used to run inside the scanner's open Aurora transaction; the subscriber's
  `SELECT WHERE finding_id` outraced the commit on warm invocations and hit the
  silent `if not finding: return` (main.py:92), so cards silently went missing.
  Now scanners accumulate pending broadcasts and fire them after
  `commit_transaction`. Affects all 4 scanners.
- `3994fdd` — added `requests` to requirements.txt (the Web-API switch added
  `import requests` but bundling didn't install it → cold-start ImportError →
  DLQ) and made imports tolerant (`from findings_subscriber import ...` fails
  at runtime because CDK bundles flat under /var/task; falls back to bare names
  — main.py:25-28).
- `839df88` — uses **Slack Web API directly, not the Slack MCP server**, for
  bot-token paths. Slack's MCP server 401s on `xoxb-*` bot tokens (only accepts
  `xoxp-*` user tokens). Documented in main.py:11-15.

## Optimization opportunities

- `soc_enrichment/features.py:30-90` — `compute_features` issues **4 separate
  Aurora Data API round-trips** per event (`_first_time_actor_on_resource`,
  `_action_rarity`, `_blast_radius_proxy`, plus the TI lookup), each a
  synchronous `execute_statement`. They all scan `events` over the same 30-day
  window for the same tenant; a single CTE/aggregate query could collapse the
  first three into one round-trip. Concrete win on p95 enrichment latency
  (spec target <30s).
- `event_router/push.py:60-69` — `send_push` calls `sns.create_platform_endpoint`
  + `sns.publish` **per device token in a loop**, synchronously. For tenants
  with many registered devices this is N×2 SNS calls in the router's hot path.
  Endpoints could be cached (token→EndpointArn) since `create_platform_endpoint`
  is idempotent but still a network call.
- `events_list/main.py:104-117` — the list handler runs a **second COUNT(*)
  query** under the same filters for every page request to populate `total`.
  Acceptable for a home-screen stat, but a windowed `COUNT(*) OVER ()` in the
  main query would halve the round-trips.

## Gotchas / non-obvious behavior

- **`event_router/push.py` and `event_router/spend_cap.py` are intentional
  copies/symlinks of shared code.** push.py mirrors `_shared/push.py` (router
  is a flat-zipped `fromAsset` Lambda with no build.sh, so `_shared/` isn't on
  its runtime path). spend_cap.py is a **symlink** into soc_enrichment. Edit
  with care — keep in sync.
- **One DynamoDB table, two counters.** `soc_llm_spend_daily` holds both the
  router's hourly push-rate counter (`push_count:` sort key) and enrichment's
  daily LLM spend (`llm_spend:` sort key). Don't assume one purpose.
- **Enrichment is drift-only.** The router only enqueues for `kind=='drift'`
  (main.py:125). GuardDuty/Inspector/SecurityHub alerts get an `events` row and
  push but no AI narrative. A spec proposing "enrich all alerts" is an
  extension, not a fix.
- **The router never raises** (main.py:136-140) — malformed events are dropped,
  not retried, because EventBridge retry won't fix bad data.
- **Dedup is per (tenant, source, source_event_id)** and only when
  source_event_id is non-null (partial unique index). Sources without a stable
  id (`_source_event_id` returns None) are never deduped.
- **`findings.frameworks` is a JSONB object, not an array** (CLAUDE.md gotcha);
  `_frameworks_to_list` defensively handles both but the canonical shape is the
  object. `findings` has **no `scanner` column** — `domain` is the proxy.
- **kill switch fails OPEN** — a flaky SSM read returns "enabled" so alerts are
  never silenced by infra flakiness. The real kill is per-tenant
  `tenant_bot_connectors.autonomous_rule_enabled`.
- **Cognito federated subject:** both `_resolve_tenant_id` and `_resolve_user_id`
  use `identities[0].userId` then fall back to `sub`, JOINing `users.sso_subject`
  — reaching for `claims.sub` directly silently 401s every Microsoft/Google
  user (CLAUDE.md canonical pattern).
- **TI feeds + NIST/OWASP packs already exist.** A spec proposing "add a TI
  feed framework" should first check: abuse.ch (Feodo+ThreatFox), KEV, Tor, and
  on-demand GreyNoise are already wired into `threat_indicators` and surfaced in
  enrichment narratives via `features.ti_matches`.
