## How to use this before proposing work

1. **Grep the register** (§ "Already Exists — Don't Rebuild" below) for your
   feature's nouns. If it's listed, your spec describes a *modification*, not
   an addition. Reframe.
2. **Open the subsystem doc** for the area you're touching (table below). It
   has the file-by-file breakdown, the data it reads/writes, and the gotchas.
3. **Verify against live code** — these docs are a snapshot. `grep -rn` the
   symbol/table/route before you commit to "new". Memory ≠ code; this map ≠ code.

## System shape (one paragraph)

One CDK app in `platform/` synthesizes **8 stacks** to a single AWS account/region.
An **API Gateway REST API** (stage `v1`, Cognito-authed) fronts ~22 Lambdas. Customer
**posture scans** run as **ECS Fargate tasks** (AWS / Azure / GCP) plus container Lambdas
(Entra, AI) — all writing into one **unified `entities` / `edges` / `findings` graph** in
Aurora PostgreSQL (Data API, no ORM). Real-time **security events** fan through one
EventBridge bus → router → enrichment (LLM-narrated) → Aurora. **CRITICAL findings**
auto-broadcast to Slack and push to iOS via APNs. Two clients — a **React SPA** (chat-first
`/` surface + legacy module surface) and a **SwiftUI iOS companion** — talk *only* to API
Gateway. **"AI security is a lens, not a silo"**: AI findings live in the same tables as cloud
findings and carry the same compliance frameworks.

### Genuine gaps (these are real new-build candidates)
- **No GitHub webhook / auto-scan-on-push** — AI scans are user-initiated only.
- **AI *code* findings carry no framework mapping** (`frameworks={}`); only cloud-side AI checks are mapped.
- **No `GET /findings/{id}` endpoint** — web + iOS work around it with list-and-filter (misses findings past the first 100).
- **GCP scanner has no AI pass** (Azure does).
- **Team-handoff** half of the iOS companion vision is under-built (the only `ShareLink` is in dead code).
- **MCP connectors beyond Slack** (Atlassian / Google / M365) are stubs.

## Cross-cutting facts a spec author keeps needing

- **Unified graph, not per-domain tables.** Write through `unified_writer.commit_scan`
  (entities/edges/findings UPSERTs). The legacy `assets` / `ai_assets` / `ai_relationships`
  tables are **dead** — don't write to them.
- **`scans` ≠ `ai_scans`** — two different tables, different PKs (`scan_id` vs `id`),
  different status enums (`completed` vs `success`). Served by different handlers.
- **Cognito subject extraction** always uses `identities[0].userId` first, then `sub`.
  Reaching for `claims.get("sub")` directly silently 401s every federated user.
- **EventBridge CloudTrail rules filter on `detail-type`, never `source`** (real events
  arrive `source: aws.<service>`).
- **Scanner `app/` copies of `scanner_core` / `ai_scanner` / `_shared` files are
  build-time generated and `.gitignore`d** — edit the source in `scanner_core/` etc.,
  not the runner copy.
- **Schema gotchas** (full list in [02](02-database.md)): `findings` PK is `finding_id`;
  `findings.status` has no `open`; `findings.frameworks` must be a JSON **object** `{}`
  (iOS decoder rejects arrays); extra JSON lives in `evidence_packet`; `entities` PK is
  `id`; `edges` uses `source_entity_id` / `target_entity_id`.
- **Admin connector routes live OUT of CDK** (bootstrap script) due to the CFN
  500-resource-per-stack cap.
- **Streaming chat is a Lambda Function URL** (LWA), invisible to an API-Gateway route grep.
