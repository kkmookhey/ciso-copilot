## Overview

The web frontend is a Vite + React 18 + TypeScript + Tailwind SPA, deployed to
S3 + CloudFront. Two surfaces coexist under one React Router tree
(`web/src/App.tsx`):

1. **Chat-first surface** at `/` — a four-column agentic chat UI
   (`web/src/chat/`) with text streaming (SSE via Lambda Web Adapter) and
   WebRTC voice (OpenAI Realtime). Auth-gated inside `ChatShell` itself.
2. **Legacy module surface** — the classic sidebar-chrome views (Dashboard,
   Findings, Risks, Policies, AI, SOC, Settings, etc.) nested under a single
   `Shell` route that supplies the auth gate + `ModuleRail` sidebar.

All authenticated API traffic goes through `web/src/lib/api.ts` (REST via API
Gateway) and `web/src/chat/chatApi.ts` (conversation CRUD via API Gateway +
streaming via the LWA Function URL). Auth is Cognito OAuth authorization-code
in `web/src/lib/cognito.ts`; tokens live in `localStorage`. The app **never**
calls upstream sources directly — only API Gateway and (for voice) OpenAI's
SDP endpoint with an ephemeral key minted by the backend.

Env boundary: `web/src/lib/env.ts` requires `VITE_API_BASE_URL`,
`VITE_APP_DOMAIN`, `VITE_STREAM_BASE_URL` at module load (throws if missing).
`apiBaseUrl` already includes the `/v1` stage prefix, so all `api.ts` paths are
relative to that. The streaming client re-adds `/v1` explicitly
(`chatApi.ts:128`).

## C) Chat / agentic UI (`web/src/chat/`)

Four-column layout assembled in `chat/Shell.tsx` (`ChatShell`):
`ModuleRail` | `ConversationRail` | `ChatCenter` | `SourceSideSheet` +
floating `FloatingChrome`.

- **`Shell.tsx` (`ChatShell`)** — auth gate (mirrors `routes/Shell.tsx`),
  conversation boot (reopen most-recent <24h old, else create new + fetch
  morning briefing), text send via `chatApi.streamMessage`, voice toggle via
  `VoiceClient`. Side-effect handling: `navigate_to` calls `nav()`;
  `filter_findings_view` is currently only `console.log`'d (`Shell.tsx:208-216`
  — wiring incomplete).
- **`ConversationRail.tsx`** — conversation list w/ inline rename, kebab menu
  (rename/delete), new-conversation button. Pure UI, props-driven.
- **`ChatCenter.tsx`** — message stream + composer + voice button host
  (imported only by `Shell`).
- **`MessageStream.tsx`** — renders message bubbles + artifact cards.
- **`Composer.tsx`** — text input + voice toggle.
- **`ModuleRail.tsx`** — left nav (brand block + nav items + email/sign-out);
  shared by both chat and legacy `Shell`.
- **`SourceSideSheet.tsx`** — right slide-over opened by a global
  `window` `"open-source-sheet"` CustomEvent (dispatched by `CitationChip` and
  entity rows). Fetches entity via `api.getEntity` or finding via
  `listFindings`+find (no by-id endpoint), renders detail.
- **`state.ts`** — `chatReducer` + `initialState`. Actions: `load`, `append`,
  `appendTool`, `streamDelta`, `streaming`, `setTitle`, `voiceUpsert`
  (`state.ts:18-43`). `voiceUpsert` keys messages by Realtime `item_id`.
- **`turnQueue.ts`** — `TurnQueue` class: background-persists sealed voice
  turns to the conversation; survives disconnect; `flushHeadOnUnload` for
  `beforeunload` (`turnQueue.ts:84`).
- **`chatApi.ts`** — see §B.

### Voice (`chat/voiceClient.ts`)

`VoiceClient` class — browser WebRTC client for OpenAI Realtime GA.
- Mints ephemeral key via `POST {apiBaseUrl}/conversations/{id}/voice`
  (body carries `toRealtimeTools(TOOLS)`).
- `getUserMedia` (echoCancellation + noiseSuppression + autoGainControl) →
  `RTCPeerConnection` w/ mic track + remote audio sink + `"oai-events"` data
  channel; SDP offer/answer POSTed directly to
  `https://api.openai.com/v1/realtime/calls` with `Bearer ek_…`.
- State machine: `off` → `connecting` → `on` (`VoiceState`, line 55).
- Handles transcript events keyed by `item_id` (out-of-order safe), barge-in
  via `cancelAssistantResponse()` (`response.cancel`), tool calls
  (`response.function_call_arguments.done` → `executeTool` → `function_call_output`),
  and `response.create` lifecycle guarding (avoids the "active response in
  progress" 400 noted in CLAUDE.md).
- Seals each turn into a `SealedTurn` and enqueues to `TurnQueue` for
  background persistence; `beforeunload` flush of the head turn.
- `shouldDropUserTranscript()` (line 87) drops empty/whitespace Whisper
  hallucinations.

### Artifact renderers (`chat/artifacts/`)

`Artifact.tsx` switch (`Artifact.tsx:34`) maps each `hint.kind` → component:
`KpiCard`, `EntityList`, `FindingCard`, `RiskCard`, `ChartBar`, `ChartDonut`,
`SeverityBreakdown`, `ApprovalCard`. `CitationChip` (`Artifact.tsx:12`)
dispatches `"open-source-sheet"`.

**`ApprovalCard.tsx`** is the only stateful artifact: full state machine

template keys fall back to `access_control`.

## Dead code (suspected)

Confidence + evidence from `grep -rn` across `web/src` (excluding `*.test.*`).

1. **`tools.ts` `chart_bar` artifact kind + `artifacts/ChartBar.tsx`** —
   **Med-High confidence.** The `chart_bar` hint type exists (`tools.ts:71`)
   and is wired into the renderer (`Artifact.tsx:39`, `ChartBar.tsx`), but
   **no tool in the 12-tool catalog emits a `chart_bar` hint** (grep for
   `kind: "chart_bar"` outside the type def returns 0 producers). The server
   tool mirror could theoretically emit one, but the client catalog never
   does. Evidence: `grep -rn 'chart_bar' web/src` → only type def + renderer.
   Likely a spec'd-but-unused artifact kept for forward-compat. Keep, but
   know it's currently unreachable from client tools.

2. **`api.ts` functions with zero non-test callers** — **Med confidence.**
   `findingsRollup` (api.ts:605), `listAIScans` (api.ts:643),
   `getEntityGraph` (api.ts:662), `getEntityRelationships` (api.ts:664),
   `revokeAIConnection` (api.ts:632). Grep across `web/src` (non-test) finds
   no call site for these five. The backend endpoints exist; the UI just
   doesn't use them yet. (`getAIScan`/`startAIScan`/`listAuthorizedRepos` ARE
   used by `RepoPicker`; `getEntity`/`listEntities` are used.) These are
   "API surface ahead of UI," not strictly dead — a future feature may wire
   them. Don't delete without confirming no in-flight work.

No fully-dead **components or routes** were found: every component file has a
non-test importer that traces to a routed surface, and every route in
`App.tsx` resolves to an imported component.

## Optimization opportunities

1. **`SourceSideSheet` + `get_finding` tool fetch up to 100 findings to
   resolve one by id** (`SourceSideSheet.tsx:184`, `tools.ts:399-401`). Every
   citation-chip click on a finding pulls `listFindings({limit:100})` and
   filters client-side. A real `GET /findings/{id}` endpoint would cut this to
   a single-row fetch and remove the silent "not in first 100" failure mode.
   Concrete and load-bearing (two call sites already comment on it).

2. **`Dashboard` fires 5 sequential-ish API calls on mount** (`Dashboard.tsx:49-58`:
   `listConnections`, `listEvents` ×2, `complianceSummary`, `findingsSummary`).
   They're independent `.then()` chains, not batched — fine, but the two
   `listEvents` calls could be one call + client-side filter, saving a round
   trip.

3. **`api.ts` `call<T>` calls `validIdToken()` on every request**, which on an
   expired token triggers a token refresh per-call with no in-flight
   de-duplication (`cognito.ts:73`). Concurrent calls (e.g. Dashboard's 5)
   each independently hit the refresh path when the token is stale. A shared
   in-flight refresh promise would collapse N refreshes into 1.

## Gotchas / non-obvious behavior

- **Two API clients, two base URLs.** `lib/api.ts` (REST) and `chat/chatApi.ts`
  (REST + STREAM). `apiBaseUrl` already includes `/v1`; the streaming call
  re-adds `/v1` literally (`chatApi.ts:128`) because it targets the LWA
  Function URL, not API Gateway. Don't "fix" the apparent double-prefix.
- **No `GET /findings/{id}`.** Single-finding lookups are list+filter. See
  Optimization 1. A spec proposing it is adding a real missing endpoint.
- **`chart_bar` artifact is defined but no tool produces it.** Don't assume the
  chat can render a bar chart today.
- **`filter_findings_view` side-effect tool is a stub** — `Shell.tsx:208-216`
  only `console.log`s it; only `navigate_to` is fully wired. "Show only
  critical AWS findings" via chat won't actually filter the findings view yet.
- **`/settings` is reachable by URL but absent from `ModuleRail` nav.**
  (`ModuleRail.tsx:5-18`.) Settings tabs Profile/Team/Billing are placeholders;
  the Cloud tab just links back to `/connect`.
- **Auth gate is duplicated**, not shared: `routes/Shell.tsx`, `chat/Shell.tsx`,
  and `components/DeepLinkGate.tsx` each re-implement the `isSignedIn()` →
  `api.me()` → status-check pattern. A change to gating logic must touch all
  three.
- **Tokens in `localStorage`** (`cognito.ts:49-67`) — XSS-exposed by design;
  the httpOnly-cookie hardening is a known deferred follow-up (CISOBrief-v2.md
  §15).
- **The client `chat/tools.ts` is a mirror of the server
  `lambda/chat_session/tools_dispatch.py`.** The voice path executes tools in
  the browser; the text/SSE path executes them server-side. Changing one tool's
  contract requires updating both to keep voice and text consistent.
- **Sign-in is email-first**, not a generic Cognito hosted-UI button:
  `discoverTenantAndSignIn` posts the email to `/auth/discover-tenant` and the
  backend returns a per-tenant IdP-hinted authorize URL (`cognito.ts:109`).
- **`ApprovalCard` idempotency:** the `approval_id` UUID generated at
  propose-time (`tools.ts:570`, `:611`) is passed as `source_approval_id` on
  create so re-approving the same card won't duplicate the risk/policy.
- **Test files** (`*.test.tsx` / `*.test.ts`) exist alongside chat, scan, soc,
  and route files (e.g. `tools.test.ts`, `voiceClient.test.ts`,
  `ApprovalCard.test.tsx`, `useScanStatus.test.tsx`, `Soc.test.tsx`). Not
  deep-documented here. Note: web lint baseline is already dirty (~42
  pre-existing errors per project memory) — that is NOT a finding.
